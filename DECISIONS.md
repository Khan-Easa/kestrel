# DECISIONS.md

A record of meaningful design decisions made while building Kestrel. Grouped by phase. Each entry covers what was chosen, what alternatives existed, and why this option won.

This file is maintained alongside the code. If a decision is later superseded, the original entry stays (with a "Superseded YYYY-MM-DD by …" note) — decisions are a history, not a snapshot.

---

## Phase 1 Decisions (Foundation)

### Web framework: FastAPI

Chose FastAPI over Flask, Django REST Framework, and Starlette-direct. FastAPI was picked because (a) it is async-native (Starlette + uvicorn under the hood), which the executor pipeline needs to interleave subprocess I/O with HTTP request handling without spawning threads; (b) it integrates pydantic for request/response validation directly in the route signature, eliminating a layer of hand-written serialization code; (c) its `Depends(...)` system gives us first-class dependency injection, which we lean on heavily for auth, settings, and the executor abstraction; (d) it generates OpenAPI docs for free, which is useful for an HTTP service that other systems will integrate with. Flask was rejected because it is sync-first and async support is bolt-on; DRF was rejected because Django is too heavy for a single-purpose service.

### ASGI server: uvicorn (with `--factory`)

Chose `uvicorn[standard]` and the `--factory` flag (`uvicorn kestrel.app:create_app --factory`). Uvicorn is the canonical production ASGI server pairing for FastAPI. The `--factory` flag means uvicorn calls `create_app()` itself rather than importing a pre-built `app` object — this keeps the FastAPI app construction inside a function (the factory pattern) so module import doesn't trigger startup work. Hypercorn and Daphne were not seriously considered; uvicorn is the standard and supports all the features we need.

### Configuration: pydantic-settings + `lru_cache` singleton

Chose `pydantic-settings` (a pydantic-v2 sibling library) with a `Settings(BaseSettings)` class and a module-level `get_settings()` wrapped in `@lru_cache(maxsize=1)`. All keys are env-var-driven with the prefix `KESTREL_` (e.g. `KESTREL_DEV_API_KEY`, `KESTREL_EXECUTOR_BACKEND`). Alternatives considered were ad-hoc `os.environ.get(...)` calls, `python-decouple`, or a YAML config file. Pydantic-settings won on three points: defaults declared next to the type (no separate "config schema" file), validation on parse (so a malformed env var fails fast at startup, not deep in a handler), and easy override in tests via `app.dependency_overrides[get_settings]`. Cached because settings shouldn't be re-read mid-process.

### Logging: structlog with stdlib bridge + uvicorn redirect

Chose structlog over the stdlib `logging` module alone. Two renderers based on `KESTREL_LOG_JSON`: `JSONRenderer` for production (one JSON object per line, ingestible by log aggregators), `ConsoleRenderer` with colors for local development. The pipeline includes a stdlib bridge (`structlog.stdlib.ProcessorFormatter`) so that `logging.getLogger(...)` calls from third-party libraries (uvicorn, FastAPI internals) flow through the same renderer — there is exactly one handler on the root logger and exactly one place that formats output. Uvicorn's three loggers (`uvicorn`, `uvicorn.error`, `uvicorn.access`) have their handlers cleared and `propagate=True` so their lines come through our pipeline too. Plain stdlib was rejected because structured key/value logging is a hard requirement for a service that needs to correlate request_id across log lines.

### Logging payload policy: never log code, only metadata

The `/execute` endpoint logs `code_length=len(req.code)` but never `req.code` itself. Same rule applies to any field that might carry secrets the caller pasted. The reason is operational: log files end up in aggregators, search indexes, and backups; logging payloads is how secrets escape into systems where they shouldn't be. Codified as a project-wide rule.

### Authentication: bearer token via `HTTPBearer(auto_error=False)` + `secrets.compare_digest`

Chose a single bearer-token scheme. The expected token comes from `KESTREL_DEV_API_KEY`. The auth check is `secrets.compare_digest`, not `==`, to defeat timing attacks (constant-time comparison). The `HTTPBearer` security scheme is constructed with `auto_error=False` so it returns `None` on missing header instead of raising — that lets us own the policy decision and return the correct status (401, not the framework's default 403). When `dev_api_key == ""` (the default), auth is bypassed entirely; this is "Phase 1 dev-mode" behaviour and Phase 7 will replace it with a real key store. Alternatives — JWT, mTLS, OAuth2 — were rejected for Phase 1 as overkill for a private/dev service. The empty-key-bypass pattern means local development needs no token while production simply sets the env var.

### Endpoint shape: `POST /execute` (auth-gated) + `GET /health` (public)

`POST /execute` carries a JSON body with the user's code, returns the executor's structured response. `GET /health` returns `{"status": "ok"}` with no auth — public on purpose, because liveness probes (load balancers, Kubernetes, monitoring) typically have no credentials. Auth-gating `/health` would either force monitoring tools to handle keys (operational pain) or cause every probe to 401 (false outage signals). The split — public diagnostic endpoints, gated work endpoints — is the rule going forward.

### Phase 1 subprocess invocation: `python -I -c <code>` (argv passing)

The Phase 1 `SubprocessExecutor` passes user code as a `-c` command-line argument: `asyncio.create_subprocess_exec(sys.executable, "-I", "-c", code, ...)`. This was the simplest end-to-end path and was deliberately a Phase 1 stub — the goal was to get the HTTP-to-runner pipeline working, not to settle the long-term code-delivery design. The known argv-passing tradeoffs (argv size cap on Linux ~128 KB, code visible to anyone with `ps` access on the host) were acceptable for a stub since Phase 2 was always going to replace this with the container backend that uses a different delivery mechanism (bind-mounted tempfile — see the Phase 2 entry). The `-I` ("isolated mode") flag was kept regardless: it disables `PYTHONPATH`/`PYTHONHOME`/user-site so user code can't influence import resolution via parent env vars.

### Output capture strategy: per-stream byte cap with truncation flag

Each stream (stdout, stderr) is read in 8 KiB chunks; once the running total exceeds `KESTREL_EXECUTE_OUTPUT_CAP_BYTES` (default 64 KiB), the buffer is sliced down to the cap and a `*_truncated=True` flag is set in the response. We do NOT raise on overflow; we return whatever fit plus the flag. Alternatives were (a) raise / 413 the request, (b) buffer everything and let memory grow. Both were rejected because the caller (and the user code) can produce unbounded output, and a code-execution service that randomly errors on chatty programs is hostile. The flag lets the caller decide whether truncation matters for their use case.

### Timeout strategy: soft contract — kill, don't raise

`asyncio.wait_for(...)` with `KESTREL_EXECUTE_TIMEOUT_SECONDS` (default 5.0). On expiry: kill the subprocess, set `timed_out=True`, set `exit_code=-1`, return an empty stdout/stderr, and respond `200 OK` with that body. We do NOT raise, do NOT 504, do NOT propagate `asyncio.TimeoutError` to the caller. The reasoning: a hung user program is not a server error — it is normal, expected output for a code-execution service, so timeout is data, not an exception. The caller decides what `timed_out=True` means for them.

### Request tracking: `X-Request-ID` middleware + structlog contextvars + `perf_counter` timing

A single ASGI middleware in `app.py` runs around every request. It (a) reads or generates an `X-Request-ID` header, (b) clears + binds structlog contextvars (request_id, method, path) so every log line in the request inherits them, (c) times the request with `time.perf_counter()` (monotonic — `time.time()` can move backwards on clock adjustments), (d) logs `request_started` and `request_finished` (or `request_failed` with traceback if the handler raises and re-raises), (e) echoes `X-Request-ID` back on the response. Standard distributed-tracing pattern; useful immediately and load-bearing once we have multiple services.

### Request size cap: `code` limited to 100,000 characters

`ExecuteRequest.code` carries `max_length=100_000` (`min_length=1` rejects empty submissions; pydantic returns 422 on either bound violation). 100 K characters is a deliberate upper bound chosen as a sane round number — large enough to comfortably accept any "reasonable" submission (a self-contained script with embedded data, a small multi-function module), small enough to prevent multi-megabyte payload abuse. No specific memory-budget calculation drove the exact value; the goal was just "have *some* cap so the endpoint can't be used as a memory-pressure vector by clients sending arbitrarily large bodies." The cap is enforced by pydantic at parse time, before the handler runs, before the executor allocates anything.

### Validation: pydantic models for request and response shapes

`ExecuteRequest` has one required field, `code: str`, with `min_length=1` and `max_length=100_000`. `ExecuteResponse` has typed fields with sensible defaults so downstream code never has to handle a missing key. Pydantic gives us 422-on-malformed-input for free — no hand-rolled validation in handlers. The `response_model=ExecuteResponse` annotation on the `/execute` route both validates outgoing data at runtime (catching schema drift) and strips fields the model doesn't declare (a built-in safety net against accidental data leaks).

### Executor input flags: subprocess invoked as `python -I -c <code>`

The Phase 1 `SubprocessExecutor` invokes Python via `sys.executable` with the `-I` ("isolated mode") flag. `-I` disables `PYTHONPATH`, `PYTHONHOME`, and the user-site directory — meaning user code cannot influence import resolution by setting env vars in the parent or by writing to `~/.local/lib/...`. This narrows the blast radius even at the subprocess stage, before Phase 2's container isolation lands.

### Test strategy: real subprocesses, no mocks; pytest-asyncio in auto mode

Phase 1 tests use FastAPI's `TestClient` with the real `SubprocessExecutor`, no mocking of `asyncio.create_subprocess_exec`. A subprocess invocation costs ~40 ms — fast enough that the integration test suite stays under 10 s, and slow enough is not a problem in exchange for the certainty that the actual subprocess pipeline works. `asyncio_mode = "auto"` in `pyproject.toml` removes the need for `@pytest.mark.asyncio` on every async test. `pytest-asyncio` was the only serious choice; `anyio`'s pytest plugin was considered but pytest-asyncio is more standard.

### Build backend: `hatchling`

`pyproject.toml` uses `hatchling` (`build-backend = "hatchling.build"`) as the PEP 517 build backend. Hatchling was chosen as the scaffolding default — it is what the project initializer produced and there was no specific reason to override it. Hatchling is a reasonable fit on its merits (modern PEP 517 / PEP 621 native, no `setup.py` required, light footprint) but the active reasoning was simply "take the default and move on." Alternatives — `setuptools` (heavier legacy default), `poetry-core` (tied to poetry's CLI workflow we aren't using), `flit-core` (more minimal but less common), `pdm-backend` (tied to pdm) — were not actively evaluated. If a future need arises (custom build hooks, version sourcing from a `__version__.py`, etc.), the backend can be swapped — it touches one stanza in `pyproject.toml`.

### Package manager: `uv`

Chose `uv` (the Rust-implemented Python package/project manager from Astral) for dependency resolution, locking, virtualenv management, and script running. Two drivers: speed — uv's resolver is dramatically faster than `pip` + `pip-tools` or `poetry`, which compounds over hundreds of `sync` / `add` operations across a long project — and single-tool ergonomics: uv replaces the `pip` + `venv` + (optional) `pip-tools` triad with one binary, so `uv sync`, `uv add`, `uv run pytest`, and `uv run uvicorn ...` all flow through the same tool. Alternatives considered: pip + venv directly (slow resolver, no lock file by default), pip-tools (still slow, three-tool stack), poetry (heavier, opinionated, slower locks), pdm (capable but smaller ecosystem). The project rule going forward: never invoke `python` / `pytest` / `uvicorn` directly — always `uv run ...` — so the venv that ships with the lockfile is the only one in use.

### Python version pin: `>=3.11,<3.12`

`pyproject.toml` declares `requires-python = ">=3.11,<3.12"` — the project targets the 3.11 series exactly. 3.11 was the latest version installed and validated at the time the project was scaffolded; the upper bound was added defensively so a future Python install doesn't silently change the runtime out from under us. Loosening the ceiling (e.g. to allow 3.12) is treated as a deliberate decision — re-run the test suite on the newer version, then bump the pin — rather than a passive one. Alternatives considered: leave the ceiling open (`>=3.11`), or pin even tighter to a single 3.11.x release. Open-ceiling was rejected for the silent-upgrade reason; tighter-than-minor was rejected as needlessly fragile (3.11.x patch releases are safe by Python's policy).

### Package layout: `src/` layout (not flat)

Source code lives under `src/kestrel/` and tests live under `tests/`. Hatchling packages `src/kestrel`. The src layout ensures imports work the same way during development and after install — `from kestrel import ...` always goes through the installed package, not the working tree. Flat layouts (where `kestrel/` is at the repo root) silently make the working tree importable, which can mask packaging bugs until install-time.

## Phase 2 Decisions (Docker isolation)

### Executor abstraction: `Protocol` (PEP 544) with `runtime_checkable`, structural typing

Defined `Executor` as a `typing.Protocol` with one method, `async def run(code, settings) -> ExecuteResponse`. `SubprocessExecutor` and `DockerExecutor` both satisfy this contract structurally — neither inherits from `Executor`. The route handler depends on `Executor`, never on a concrete class. Alternatives were `abc.ABC`/`abstractmethod` (nominal subtyping — implementations must inherit) or no abstraction at all (call the concrete class). Protocol won because (a) it lets us write fakes in tests with no inheritance ceremony, (b) implementations don't have to know they conform to the contract, which keeps coupling low, (c) `runtime_checkable` lets us assert conformance with `isinstance(x, Executor)` if we ever need to.

### Backend selection: `KESTREL_EXECUTOR_BACKEND` Literal env var + `lru_cache`d `get_executor`

`Settings.executor_backend: Literal["subprocess", "docker"]` selects the implementation. `get_executor()` is `@lru_cache(maxsize=1)` so the executor is instantiated once per process and reused. The route handler asks for it via `Depends(get_executor)`. In tests, `app.dependency_overrides[get_executor] = lambda: SubprocessExecutor()` (or `DockerExecutor()`) swaps the backend without touching env vars or restarting the process. Default is `"docker"` since Phase 2 — `"subprocess"` is preserved as a fallback for hosts without Docker (and as the test path for the parametrized integration suite).

### Container-per-request, no warm pool

Each `/execute` call spawns a fresh container, runs the user's script, and lets `--rm` reap it. There is no warm-container pool, no reuse, no per-tenant allocation. Two reasons drove this. **Security simplicity:** by construction, no state can leak between requests — no concern about "did the last user leave something running?", no per-tenant cleanup tracking, no risk of one tenant's env leaking into another's. The isolation story is straightforward to defend. **Phase-appropriate:** Phase 2 is about getting isolation correct, not optimizing throughput. A warm pool would add real complexity (pool sizing, eviction, container reuse review, health checks, leaked-fd/process/`/tmp`-state tracking) that is not justified at this phase. The accepted cost is ~300–800 ms cold-start per request; pool optimization can come later if benchmarks show it is needed. Locked design decision — do not relitigate without a strong reason.

### Code delivery: bind-mounted tempfile at `/sandbox/main.py:ro`

The user's code is written to a host tempfile (in a per-request `tempfile.TemporaryDirectory(prefix="kestrel-")`), bind-mounted into the container as read-only at `/sandbox/main.py`, and executed via `python /sandbox/main.py`. We do NOT pipe the code via stdin or pass it as `python -c "<code>"`. Three benefits drove the choice, all of them desirable: (1) **Real tracebacks** — Python reports errors as `/sandbox/main.py:NN` instead of `<stdin>:NN`, so the caller sees a real file path and line number that can be mapped back to the submitted source. (2) **No argv leakage** — code is not visible to `ps` on the host or under `docker top` / `docker inspect`'s container metadata, unlike `python -c "<code>"`. (3) **Symmetry with normal script execution** — one canonical path ("Python runs a file") instead of a special "Python reads stdin" or "Python parses argv" mode, which makes the executor easier to reason about. Read-only mount (`:ro`) prevents the user code from rewriting itself mid-execution. Locked design decision — do not relitigate without a strong reason.

### Docker integration: CLI via `asyncio.subprocess`, not `docker-py`

The `DockerExecutor` shells out to the `docker` CLI using `asyncio.create_subprocess_exec("docker", "run", ...)`. We do NOT import `docker-py` (the official Python SDK). Three reasons combined to drive this. **Fewer dependencies:** keeping `pyproject.toml` lean is itself a goal — one fewer pinned library is one fewer thing to update, audit, and break on. **CLI flag familiarity:** the `docker run` flags in code read the same as a shell command (`--network none --read-only --user 65534:65534 ...`), so reasoning about isolation policy doesn't require translating between shell vocabulary and SDK argument names. The same flags you would test interactively are literally the ones in code. **Avoiding SDK pin pain:** `docker-py` versions and Docker Engine versions have historically had fragile compatibility around new flags and API changes; pinning to a CLI binary on PATH is a more stable contract because the CLI's flag surface is documented in `man docker-run` and changes slowly. Tradeoffs accepted: we depend on the `docker` binary being on PATH at runtime, and we parse CLI stdout for output capture (handled by the same `_read_stream` helper Phase 1 used for the subprocess executor). Locked design decision — do not relitigate without a strong reason.

### Container naming: `kestrel-exec-<uuid-hex>` prefix

Every container is launched with `--name kestrel-exec-<uuid4-hex>`. The deterministic prefix is what makes the orphan sweep work — `docker ps -aq --filter name=kestrel-exec-` finds exactly Kestrel's containers and nothing else. The UUID suffix prevents name collisions when many requests arrive concurrently. Without the prefix, the sweep would either miss orphans (too narrow a filter) or risk killing unrelated containers (too broad).

### Orphan invariant: `docker kill` on timeout + sweep on startup

Killing the `docker run` CLI process does NOT stop the container — the CLI is just a client; the real container parent is `containerd-shim`. So on timeout, `DockerExecutor` issues an explicit `docker kill <name>`. Symmetrically, when the app starts, `sweep_orphan_containers()` lists and force-removes any `kestrel-exec-*` containers left behind by a previous crashed run. Both halves are required: per-request kill prevents leakage during normal timeouts; startup sweep handles the case where the server itself crashed mid-request. The sweep is best-effort — if the Docker daemon is unreachable, it logs `orphan_sweep_skipped` and continues, so the app still starts on subprocess-only hosts.

### Container hardening flags (the bundle)

Every container runs with: `--rm` (auto-remove), `--network none` (no network namespace at all — not just firewalled, *absent*), `--read-only` (rootfs immutable), `--tmpfs /tmp:size=64m` (only writable area, capped), `--user 65534:65534` (UID/GID `nobody`, no privileged identity inside), `--memory 256m --memory-swap 256m` (memory cap with `--memory-swap == --memory` to disable swap so OOM-kill is the actual ceiling, not a slow death by paging), `--cpus 1.0` (CPU quota), `--pids-limit 64` (fork-bomb defence). These are passed as separate argv tokens (not concatenated) so the CLI parses them unambiguously. The set is applied in the same order on every call — there is one canonical command, not flag composition logic.

### `/tmp` tmpfs at 64 MiB

`--tmpfs /tmp:size=64m` is the *only* writable filesystem available to user code. It is RAM-backed (tmpfs), so the size cap is what prevents a malicious script from filling host RAM by writing to disk. 64 MiB is a deliberate small number — enough for normal scratch use (intermediate files, pickled blobs), small enough that a runaway can't move the needle. The cap is enforced by the tmpfs filesystem driver itself (returns ENOSPC on overflow), which works on every Linux including WSL2.

### Runtime image: `python:3.11-slim` pinned by digest, non-root user baked in

`Dockerfile` is `FROM python:3.11-slim@sha256:6d85378d88a19cd4d76079817532d62232be95757cb45945a99fec8e8084b9c2`. A `sandbox` user/group is created with UID/GID 65534 (matching `nobody`), no home directory, `/usr/sbin/nologin` shell. `WORKDIR /sandbox` and `USER 65534:65534` are set in the image — defence in depth, since the runtime `--user` flag also enforces this. The base is `python:3.11-slim` (not `python:3.11`) because the slim variant is ~80 MB smaller and has fewer packages = smaller attack surface. The base image is **pinned by digest** for two reasons that the same one-line change buys at once: (1) **supply-chain integrity** — a compromised registry or malicious actor cannot substitute a backdoored image under the `python:3.11-slim` tag; the SHA256 is content-addressed so only the exact bytes we originally chose will match; (2) **build reproducibility** — the same Dockerfile produces the same image months later regardless of how many times upstream re-pushes the tag, which is useful for debugging ("did the base change?"), compliance audits, and ensuring CI and dev environments build identical artifacts. Trade-off accepted: digest must be manually bumped when we want a newer base (security patches included), so we do not silently inherit upstream changes — that is the whole point.

### Default backend = `"docker"`, subprocess kept as fallback

After Phase 2 shipped, `Settings.executor_backend` default was flipped from `"subprocess"` to `"docker"`. The subprocess backend is preserved (not deleted) for two reasons: (a) hosts without Docker (CI runners, restricted environments) can still run the service by setting `KESTREL_EXECUTOR_BACKEND=subprocess`, and (b) the integration test suite parametrizes 20 black-box tests over both backends in `tests/conftest.py`, ensuring the contract is identical and any future contract drift is caught.

### Test fixture symmetry: parametrize over backends, skip docker if daemon down

`tests/conftest.py` defines `client(request)` parametrized over `["subprocess", "docker"]`, using `app.dependency_overrides[get_executor]` to install the chosen executor. The fixture calls `_docker_reachable()` (a `lru_cache`'d check that runs `docker info` once per session) and `pytest.skip(...)`s the docker variants if the daemon is down — so the suite stays green on hosts without Docker. A separate `docker_client` fixture (always docker, no subprocess variant) is used by the isolation tests, which only make sense against a real container. The override-based approach was chosen over env-var-driven backend selection because it is per-test and doesn't pollute the `lru_cache` of `get_settings`/`get_executor`.

## Phase 3 Decisions (Resource limits)

### Hardening flags: `--security-opt no-new-privileges` + `--cap-drop ALL`

Phase 3 Step 1 added two flags to the `docker run` command on top of the existing `--user 65534:65534`. **`no-new-privileges`** sets the kernel `NO_NEW_PRIVS` bit, which prevents a process from gaining additional privileges across `execve()` — setuid binaries don't escalate, file capabilities don't apply. **`--cap-drop ALL`** empties the container's effective Linux capabilities bitmap, removing every per-area root power (CAP_NET_RAW for raw sockets, CAP_SYS_ADMIN for mounts, CAP_SYS_PTRACE for `ptrace`, CAP_SYS_MODULE for kernel modules, etc. — the kernel divides root's authority into ~37 named privileges and we drop them all). Together these two flags mean that even if a setuid binary somehow exists inside the image, executing it can't restore capabilities — the kernel refuses both the setuid bit transition and the capability grant. Defence in depth: each flag covers what the other might miss. Verified at runtime by reading `/proc/self/status` inside the container — `test_isolation_no_new_privileges` asserts `NoNewPrivs:\t1`, `test_isolation_capabilities_dropped` parses the `CapEff:` hex bitmap and asserts it equals 0. Catches a regression where either flag is silently dropped from the run command.

### Pids cap: `--pids-limit 64`

Phase 3 Step 2 chose 64 as the per-container process/thread limit. The cap is the fork-bomb defence — without it, a `while True: os.fork()` loop in user code can exhaust host PID slots until the kernel itself starts refusing process creation system-wide. 64 is the chosen number because it is **large enough** to comfortably handle normal multi-threaded Python (a few worker threads, GC threads, asyncio loop threads, the parent process itself) and **small enough** that a fork bomb hits `OSError`/`EAGAIN` from the kernel within milliseconds, well before the 5-second wall-clock timeout fires. Verified by `test_isolation_pids_limit_enforced`: parent forks in a loop (capped at 200 attempts to bound the test), children `time.sleep(60)` + `os._exit(0)` to hold their pid slot, parent counts how many forks succeeded. Lenient-but-bounded assertion — `0 < forked < 200` proves the cap fired (loop didn't exhaust); `forked <= 64` catches a regression that loosens the cap (some kernel-version pid-accounting jitter is allowed within that bound).

### Seccomp: keep Docker's default profile, no custom profile

Phase 3 Step 3 considered authoring a custom seccomp profile narrowly tailored to "what user Python actually needs to run." Rejected. Docker's default profile is the right cost/benefit point. **Why:** the default already blocks ~44 dangerous syscalls (including `mount`, `unshare`, `setns`, `kexec_load`, `bpf`, `ptrace` against unrelated processes, etc.) and is maintained by Docker's security team — they update it as new kernel features land. Authoring a custom profile for *arbitrary* user Python is structurally hard: too broad (effectively matches the default — pointless work) or too narrow (breaks legitimate Python features the moment a user imports something the profile didn't anticipate — `socket`, `multiprocessing`, `os.fork`, `subprocess`, `mmap`, file I/O on uncommon filesystems, etc., all touch syscalls the default permits but a hand-written narrow profile might miss). For an arbitrary-Python execution service, the maintenance burden of keeping a custom profile in sync with both kernel changes *and* Python's evolving syscall surface outweighs the marginal security gain. Verified at runtime by `test_isolation_seccomp_filter_active`, which asserts `Seccomp:\t2` (mode 2 = BPF filter attached). The test does not try to prove any individual syscall is blocked — Docker's blocklist varies across versions, and "filter is on" is the durable signal. Catches a regression where someone adds `--security-opt seccomp=unconfined` for "debugging" and forgets to remove it.

### Tmpfs size-cap test: raw `os.open`/`os.write`, not buffered I/O

Phase 3 Step 4 added `test_isolation_tmpfs_size_capped` to verify the `--tmpfs /tmp:size=64m` cap actually fires. The test deliberately uses **raw file descriptor I/O** (`os.open` + `os.write`) instead of Python's high-level `open(...).write(...)`. Reason: the high-level API returns a buffered file object — writes don't immediately hit the kernel; Python collects them in memory and flushes when the buffer fills. For a tmpfs cap test, that delay would distort the test in two ways: the ENOSPC error wouldn't fire on the iteration that actually overran the cap (it would fire later, on the buffer-flush boundary), and the byte count we observe wouldn't reflect actual kernel-level writes. Raw fd I/O bypasses the buffer entirely; every `os.write(fd, chunk)` is a real syscall, so the cap fires at a predictable byte count. The 1-MiB chunk size is small enough that the cap's exact byte position is observable. Upper-bound assertion `50 < written_mib <= 64` catches both a regression that loosens the cap (`<= 64` would fail if someone bumps `size=64m` to `size=64g`) and one that tightens it implausibly (`50 <` rules out spurious early failures). The lower bound has slack because tmpfs accounts for filesystem metadata as well as user data — the actual usable write capacity is slightly less than the configured 64 MiB. **Crucially**, the tmpfs `size=` cap is enforced by the **tmpfs filesystem driver itself** (returns ENOSPC = errno 28 on overflow), *not* by the cgroup memory controller — which is why this test works on WSL2 even though `--memory` doesn't.

### `--memory` test with `pytest.skipif(_is_wsl2)`

Phase 3 Step 5 added `test_isolation_memory_limit_enforced` plus a private `_is_wsl2()` helper used in a `pytest.mark.skipif`. **The skip is necessary because the WSL2 kernel doesn't enforce the Docker memory cgroup** — Docker passes the flag, the cgroup is created, but the actual OOM-killer doesn't fire when processes inside the container exceed the configured limit. Without the skip, the test would falsely report "cap not enforced" on every WSL2 development host, even though production Linux *does* enforce it. The skip is the honest answer: don't claim a test result we can't verify. `_is_wsl2()` reads `/proc/version` and looks for the substring `"microsoft"` (WSL2 advertises a Microsoft-built kernel; real Linux does not). The skip reason is human-readable so pytest output is self-explanatory. On real Linux, the test allocates 1-MiB `bytearray` chunks in a loop up to 512 MiB (twice the 256 MiB cap, generous headroom — `bytearray(N)` zero-fills, which forces page faults so the cgroup actually accounts for the memory rather than counting only reserved virtual address space). Asserts `timed_out is False` (OOM-kill should fire faster than the 5-second wall-clock, sub-second on a tight allocation loop) and `exit_code != 0` (OOM-killed containers exit non-zero — typically 137 = 128 + SIGKILL=9, but we use `!= 0` rather than `== 137` for resilience to Docker's signal-reporting convention across versions). The error messages are diagnostic — if the cap silently fails and the loop completes normally, the assertion message includes the stdout (`'ALLOCATED 512 MiB'`) so the regression is immediately obvious.

### HTTP rate limiting deferred out of Phase 3

Phase 3 Step 5 considered including HTTP-layer rate limiting (per-client request-per-time-unit caps) before declaring Phase 3 done. Deferred. **Why:** Phase 3's stated theme is *per-execution* resource limits — what a single request can consume on the host (memory, CPU, pids, tmpfs, network, capabilities). Rate limiting is a different concern: how many requests *one caller* can issue per unit time, regardless of what each request does. It naturally lives in the auth/hardening layer alongside API keys, quotas, tenancy, and abuse-mitigation policy — concerns that arrive in Phase 6 (auth) or Phase 7 (production hardening) per the project's phase plan. Building rate limiting in Phase 3, before there's a meaningful identity or tenancy concept, would mean implementing it against the IP address of the client — useful but coarse, and likely to be replaced once `KESTREL_DEV_API_KEY` evolves into a real key store. Phase 3 ships without rate limiting; the decision is captured here so a future "why isn't there rate limiting?" question has the right answer ready.

## Phase 4 Decisions (Sessions)

### Communication protocol: home-grown JSON-line over stdin/stdout

Phase 4 introduces a long-running Python process inside each session container; the API needs a wire format to send code in and receive stdout/stderr/exit_code back. Chose a home-grown JSON-line protocol over the container's stdin/stdout, framed as "one JSON message per line." The container runs a small kernel script on startup that loops: read a JSON line from stdin, exec the code in a persistent global namespace, write a JSON-line response to stdout. The API holds onto the container's stdin/stdout pipes (`docker run -i ...`) for the session's lifetime and writes/reads JSON lines on demand.

Two real alternatives were considered and rejected. The Jupyter messaging protocol (ZeroMQ across five sockets per kernel, HMAC-signed multipart messages, `ipykernel` as the runtime) would have been production-grade and would have given us Phase 6's rich-output story essentially for free, but at three costs that didn't fit Phase 4: (a) ZMQ wants TCP, and we'd have had to either weaken the Phase 2 `--network none` lock or design unix-socket bind-mounts into the container, both real work; (b) `ipykernel` brings IPython's auto-magic and cell-level introspection, behaviour we may not want; (c) ZMQ + HMAC + Jupyter-specific debugging is a learning-cliff for what Phase 4's stated theme (sessions) does not actually need. A hybrid (home-grown wire transport but Jupyter-shaped message field names) was also considered — rejected as adding discipline now for migration insurance against an option we may never exercise.

The chosen design's tradeoffs are accepted explicitly: (1) Kestrel sessions will not be Jupyter-compatible — a Jupyter UI cannot attach to a Kestrel session; (2) we own every edge case ourselves (kernel crash mid-message, stdout that arrives long after the request "completes," partial output capture), but those are bounded problems we can solve incrementally. The win is that `--network none` survives Phase 4 untouched — pipes are file descriptors, not sockets, so there is no network-namespace concern at all.

### Concurrency within a session: reject second concurrent execute with HTTP 409

A session has one Python process behind it, which is single-threaded by nature. When two `POST /sessions/{id}/execute` calls arrive for the same session while one is still running, the server rejects the second with HTTP 409 Conflict and `{"error": "session_busy"}`. The caller decides whether to retry. The first request runs through normally; the rejection is immediate (no queueing, no waiting).

Two alternatives were considered. **Serialize** (queue the second behind the first) was rejected because it builds an implicit unbounded per-session work queue, and HTTP semantics make "wait an arbitrary time for someone else's code to finish" awkward — wall-clock timeout starts when, request arrival or execution start? Either answer is wrong half the time. **Cancel-and-replace** (interrupt the running execution, start the new one) matches Jupyter's "interrupt and re-run cell" UX but is too aggressive as a default; cleanly interrupting arbitrary user code is genuinely hard (signal handling, partial-state cleanup), and surprising if the contention is accidental.

Reject-with-409 won on three points: (1) every request gets a deterministic answer in bounded time; (2) 409 is the standard HTTP code for "resource in a conflicting state," well-understood by clients and proxies; (3) a normal REPL-shaped client only issues one execute at a time, so contention is a bug or a race, and 409 surfaces it loudly instead of absorbing it. Cancel-and-replace can be added later as an explicit `?force=true` opt-in if a use case appears — easier to add than to remove.

### Auth scope: bearer token + session_id as capability (unguessable UUID4)

`/sessions/*` endpoints are gated by the same bearer-token check as `/execute` (consistency with Phase 1's auth policy). Once a session exists, the session_id itself is treated as a capability: a UUID4 carries 122 bits of entropy and is unguessable in practice, and knowledge of the session_id *is* the right to access it. We do not track ownership explicitly in Phase 4 — there is exactly one bearer token (`KESTREL_DEV_API_KEY`), so multi-user differentiation has nowhere to come from at the auth layer.

The defence-in-depth posture is intentional: an attacker needs *both* the bearer *and* a specific session_id to do anything to a session. A bearer leak alone gives no session access (the attacker still doesn't know which session_ids exist); a session_id leak alone gives no API access (without the bearer, every endpoint returns 401). Same family of pattern as JupyterHub tokens, Google Docs share-links, AWS pre-signed URLs — unguessable identifiers used as capabilities.

Explicit ownership tracking (Redis-stored `session_id → bearer_hash`; even with a valid bearer you can only access sessions you created) was considered and rejected for Phase 4: there is only one bearer in this phase, so ownership has nothing to differentiate. Phase 7 brings real per-key API keys, and ownership tracking will be added then on top of the existing capability semantics — no semantic break, no migration. The design composes cleanly forward.

One operational discipline this commits us to: never log the full session_id at INFO level. Logs end up in aggregators, search indexes, and backups; the same reason we never log `req.code` applies here. Log a prefix for traceability (`session_id_prefix=abc12345`); the prefix is enough for ops correlation, not enough for an attacker to use as a capability.

### Phase 2 lock formally amended: warm pool *is* allowed for session containers (not for per-request execution)

The Phase 2 design lock "fresh container per request, no warm pool" is preserved for the original `/execute` endpoint and for any future stateless one-shot endpoints. Phase 4 introduces a separate, *session-scoped* container model where one container outlives many requests within one session, and the Phase 4 scope mandates a container pool ("Container pooling for efficiency") for warming session containers ahead of time. The two design locks address different concerns: Phase 2's lock was about *cross-request* state leakage (no state carries between independent users' executions); Phase 4's pool only serves sessions, and a session is by definition a single user's stateful conversation, so the cross-request-leakage concern does not apply within a pool. The contradiction is therefore shallow — surface-level wording, not substance — but it is worth noting explicitly so a future reader can see both decisions in context.

### Substep 2A: kernel script lives at `docker/executor/kernel.py`, not in `src/kestrel/`

The persistent REPL kernel runs *inside* the container as a separate process from the API server. It has no business knowing about FastAPI, pydantic, settings, or any other host-side machinery — it imports nothing from `kestrel.*`, only Python's standard library. Two locations were considered. Putting it under `src/kestrel/` (e.g. `src/kestrel/execution/kernel.py`) would put it next to its conceptual peers (the executors). Putting it under `docker/executor/` (next to the Dockerfile that bakes it into the image) makes the layout signal what it actually is: an artifact that ships *with the runtime image*, not with the host package. The second location won. The Dockerfile's `COPY kernel.py /sandbox/kernel.py` is the giveaway — the file's only purpose is to land inside the image. Keeping it adjacent to the Dockerfile means anyone modifying the image sees both files together; keeping it out of `src/kestrel/` means the host package's import graph is not muddled by a script that has no place in it. As a side benefit, the container image build does not need to install the `kestrel` package, which keeps the image small and the build simple.

### Substep 2B: kernel catches `Exception` only; `SystemExit` and `KeyboardInterrupt` end the session

The kernel's `_execute` helper wraps `exec(code, namespace)` in `try / except Exception`. This is the standard Python idiom: catch normal errors (`NameError`, `TypeError`, `ValueError`, custom user exceptions) and let signal-like exceptions (`SystemExit`, `KeyboardInterrupt`, `GeneratorExit`) propagate. Three options were considered. **Catch `BaseException`** would make the kernel survive `sys.exit()` and Ctrl-C — but doing so violates a strong Python convention (signals are *meant* to interrupt) and would mean a user's `sys.exit(2)` silently becomes a normal response, which is confusing. **Catch `Exception` only** is what we chose: a user `NameError` is reflected in the response and the kernel keeps running; a user `sys.exit(0)` ends the kernel and terminates the session. The user has to make a new session if they want to keep working. **Catch `Exception` plus a separate `except SystemExit:` clause** that converts `sys.exit()` into a normal response with the requested exit code was considered as a Phase 4 polish item — adds code for a rare edge case, deferred. Acceptable tradeoff: `sys.exit()` in a session is uncommon, and the failure mode (session ends with a clear `SessionTerminated`) is bounded.

### Substep 2C: image tag bumped `kestrel-runtime:0.2.0` → `kestrel-runtime:0.3.0`

Phase 4 substep 2 added a new file (`/sandbox/kernel.py`) and a new default `CMD` to the runtime image. The image content changed; therefore the tag must change. Three alternatives were considered. **Reuse `0.2.0`** would silently break anyone who pulled the old tag expecting the old behaviour — bad versioning hygiene, hard to roll back. **Jump to `1.0.0`** would imply Phase 4 is a stable major release, which it is not (Phase 4 is one of seven). **Bump the patch component to `0.3.0`** matches our existing convention (`v0.1.0` for Phase 1, `v0.2.0` was Phase 2, `v0.3.0` for Phase 3 ship). The default `Settings.executor_docker_image` was bumped in lockstep so the stateless `/execute` path uses the new image too — preserving the byte-identical hardening contract. The previous image (`:0.2.0`) is left in the local registry alongside `:0.3.0` so prior commits remain runnable; CI / production rollouts pull the tag from `Settings`, which always reflects the current default.

### Substep 3: host-side runtime client lives at `src/kestrel/execution/session_runtime.py`

The new class wrapping a `docker run -i` subprocess and the JSON-line protocol lives next to its peers in the execution package — sibling to `docker_executor.py` and `manager.py`. Two alternatives were considered. **A new top-level `src/kestrel/sessions/` package** would group session-related modules together (registry, lifecycle manager, runtime client) once they all exist, but at substep 3 there is exactly one such file; introducing a package for one file is premature abstraction. **`src/kestrel/execution/sessions.py`** as a generic name was rejected because the file specifically implements the *runtime client* (the protocol-speaking subprocess wrapper); other session concerns (registry, eviction, HTTP handlers) belong in different files at different layers. Keeping the name specific (`session_runtime.py`) leaves room to split later. If a session registry, an HTTP route, a lifecycle daemon, etc. all accumulate in `kestrel.execution`, that is the moment to extract a `kestrel.sessions` package — not now.

### Substep 3: SessionRuntime is fully async, mirroring DockerExecutor

The class exposes an async public surface: `await SessionRuntime.start(...)`, `await runtime.execute(code)`, `await runtime.close()`. Internally it uses `asyncio.create_subprocess_exec`, `asyncio.wait_for`, and `asyncio.create_task`. **A synchronous variant** (using `subprocess.Popen` + threads) was considered briefly and rejected: the rest of the project — FastAPI, the request-ID middleware, both existing executors — is async top-to-bottom, and forcing FastAPI route handlers to call into a synchronous SessionRuntime via `asyncio.to_thread()` would inject a thread per call and undo the cooperative-scheduling benefit. The async shape composes cleanly with how the route handler will call it in substep 4+. The "async constructor" idiom (synchronous `__init__` for attribute setup; `async classmethod start` for the docker spawn) is the standard pattern for libraries with required async initialization (aiohttp, asyncpg, aiomysql) — chosen so callers always get a fully-ready instance back from `await SessionRuntime.start(...)` rather than having to remember a separate readiness call.

### Substep 3: stderr is drained on a separate pipe into a bounded side-channel buffer

The container's stderr is **not** merged into stdout. Two alternatives were considered. **Merge stderr into stdout** (`stderr=asyncio.subprocess.STDOUT`) was rejected because the JSON-line protocol on stdout depends on the line stream being pure JSON — any non-JSON bytes (a Python deprecation warning, the `Traceback (most recent call last):` lines Python emits when an unhandled `BaseException` ends the process, a stray `os.write(2, ...)` from user code) would corrupt `readline().json.loads(...)` parsing and surface as a `SessionProtocolError` for an entirely unrelated reason. **Discard stderr entirely** (`stderr=DEVNULL`) was rejected because a kernel that crashes loudly should leave a forensic trail — diagnosing "session died unexpectedly" is much easier when the Python runtime's last words are still readable. Chosen design: a separate stderr pipe drained by a background `asyncio.create_task(_drain_stderr())` into a `bytearray` capped at 64 KiB. Past the cap, we keep reading and discarding (we *cannot* stop reading — that would risk the kernel blocking on a full pipe and deadlocking against our stdout reader). Surfaced read-only as the `stderr_buffer` property for diagnostics; never participates in the protocol.

### Substep 3: SessionRuntime owns per-message timeout only; idle and total-lifetime caps live at the session-registry layer

The runtime client wraps every `execute()` reply read in `asyncio.wait_for(..., timeout=self._timeout_seconds)`. On expiry, `_terminated = True`, the container is killed by name, the subprocess is reaped, and `SessionTimeout` is raised. Subsequent calls fast-fail with `SessionTerminated`. Three alternatives were considered. **No timeout in this layer** (let the HTTP layer's wall-clock timeout handle it) was rejected because the HTTP timeout is per-request, not per-execute, and a wedged execute with no per-message ceiling means an infinite wait inside the runtime client — bad shape regardless of who calls it. **Per-message timeout that interrupts but does not kill** (sentinel-based interrupt) was rejected because the kernel has no interrupt path other than `BaseException` propagation, which already ends the session anyway; the "interrupt" notion is fictitious. **The chosen design** lets the runtime own one well-defined responsibility (per-message wall clock) and pushes idle eviction, total session lifetime, and any tenancy-scoped quotas to the session-registry layer in a future substep. Each cap composes in its proper place; the runtime stays small and testable.

### Substep 4: Registry lives at `src/kestrel/execution/session_registry.py`

The in-memory session registry sits next to its peers in the execution package — sibling to `docker_executor.py`, `manager.py`, and `session_runtime.py`. Two alternatives were considered. **A new top-level `src/kestrel/sessions/` package** would group session-related modules together once they all exist (registry now, HTTP routes next, container pool later) — rejected as premature: substep 3 used the same reasoning to keep `SessionRuntime` in `execution/`, and reversing that decision for the second file in the series would be inconsistent. **A vague name like `src/kestrel/execution/sessions.py`** was rejected because the file specifically implements the *registry* (in-memory map, lifecycle, eviction); other session concerns will land in different files. The trigger to extract a `kestrel.sessions` package is "second or third session-machinery file lands and the execution/ folder is getting noisy" — at that point all of them move together. Until then, `execution/` is the right home.

### Substep 4: Idle eviction via background asyncio.create_task sweep, not lazy-on-access

A `SessionRegistry` carries a long-lived `asyncio.Task` (the sweeper) created in `start()` and cancelled in `aclose()`. The sweeper wakes every `session_sweep_interval_seconds`, takes a snapshot of the sessions dict, and closes every entry whose `last_used` is older than `session_idle_timeout_seconds`. Three alternatives were considered. **Lazy-on-access cleanup** (sweep inline on every `create`/`get`/`list`) was rejected because containers hold real OS resources (memory cap, namespaces, pids), and "evict only when somebody asks" means an idle session lingers indefinitely if nothing touches the registry — exactly the failure mode the idle threshold is supposed to prevent. **Hybrid (lazy + slow background)** was rejected as premature complexity: two mechanisms doing the same job make it harder to reason about which path evicted what. The chosen background-sweep design pairs naturally with FastAPI's `lifespan` (the task is started in lifespan-startup, cancelled in lifespan-shutdown) and emits log lines (`session_evicted_idle`) so eviction is observable. The lifecycle pair is `start()` / `aclose()` rather than `__aenter__` / `__aexit__` — FastAPI's `lifespan` is already an async context manager and nesting two contexts is awkward. `aclose()` is **terminal**: calling `start()` after `aclose()` raises `RuntimeError("registry is closed")`. The registry is built once per process, started once, closed once; restart-after-close is not a supported state and forbidding it removes a class of "is this registry healthy?" ambiguity.

### Substep 4: Two new Settings fields — `session_idle_timeout_seconds` (900s default) and `session_sweep_interval_seconds` (60s default)

Both numbers live on `Settings`, env-var-driven by `KESTREL_SESSION_IDLE_TIMEOUT_SECONDS` and `KESTREL_SESSION_SWEEP_INTERVAL_SECONDS`. Three alternatives were considered. **Hardcoded module constants** were rejected because tests need to inject very short thresholds (effectively negative, to force every session to be "expired") and monkey-patching module globals is a worse test pattern than `Settings(**defaults)`. **A single Settings field with hardcoded sweep cadence** was rejected because the sweep interval is what gates how fast tests can drive eviction — both knobs need to be test-overridable. **The chosen design** matches the existing pattern (every other knob — timeout, output cap, image tag, backend — is on Settings) and the 15-min / 1-min defaults give 15× over-sampling between sweeps, so worst-case eviction latency past the threshold is one sweep cycle. The per-execute timeout used to spawn each `SessionRuntime` reuses `execute_timeout_seconds` rather than introducing a third knob — same semantic ("max wall-clock waiting for one execute reply"), so one number is easier to reason about than two.

### Substep 4: `create()` and `list()` share a frozen `SessionInfo` dataclass; `get_runtime` bumps `last_used`, `get_info` does not

`SessionInfo` is a `@dataclass(frozen=True, slots=True)` carrying `session_id`, `created_at`, and `last_used`. Both `create()` and `list()` return it — uniform shape means HTTP-layer serialization has one type to handle, and `list()` results are type-safe. The internal `_Entry` (paired live runtime + info) is *not* frozen — bumping `last_used` is done via `entry.info = replace(entry.info, last_used=_utcnow())`, the canonical "immutable update" pattern. Two alternatives for the return shape were considered. **Just returning `session_id: str`** was rejected because the future HTTP handler needs `created_at` in the response body and would have to immediately re-query for it. **Returning the live `SessionRuntime`** was rejected because it conflates layers (the create endpoint per §6.4 is "register a session," not "register and run") and breaks symmetry with `list()` (which can't return live runtimes). The asymmetric pair `get_runtime` / `get_info` is deliberate: `get_runtime` is the path handlers use *just before calling execute()*, so touching `last_used` correctly marks the session as active; `get_info` is the read-only path used by ops listings, and bumping `last_used` there would mean an ops query resets every session's idle clock and prevents eviction forever. The single helper `_utcnow()` produces every timestamp in the file — both `created_at`/`last_used` (for display) and the idle math in `_sweep_once` — so wall-clock vs. monotonic-clock is a one-line swap if NTP jumps ever become a real concern. Wall-clock was chosen for now because at a 15-min idle scale, sub-second clock corrections are negligible and one timestamp type is simpler than two.

### Substep 4: Sweep concurrency handled by snapshot-before-iterate, not asyncio.Lock

`_sweep_once` materializes the dict's items into a list first (`list(self._sessions.items())`) and iterates that snapshot. Three alternatives were considered. **`asyncio.Lock` around all registry operations** was rejected as actively bad — the sweep holds the registry mutex while awaiting `runtime.close()` (up to 2 seconds in the graceful → forceful close ladder from substep 3), stalling every concurrent `create`/`get`/`delete` in the whole process. That defeats async. **Tolerating raw `RuntimeError: dictionary changed size during iteration` and retrying** was rejected as crude — relies on a Python error message that's not a stable API, and we lose iterator progress on each retry. **The chosen snapshot pattern** is the standard Python idiom for "iterate-and-mutate" and relies on two existing invariants: `SessionRuntime.close()` is idempotent (substep 3 contract), so a delete/sweep race that closes the same runtime twice is safe; and `self._sessions.pop(session_id, None)` with the `None` default never raises if the entry is already gone. Races are bounded and harmless — at worst, `close()` lands on a dead runtime, which is a no-op. `SessionNotFound` subclasses `KeyError` so callers can `except KeyError` if they want generic handling, or the specific class if they want the precise contract.

### Substep 5: Session routes live in a new `src/kestrel/api/sessions.py` with an `APIRouter(prefix="/sessions")`

The five `/sessions/*` endpoints land in their own file with `APIRouter(prefix="/sessions", tags=["sessions"])`, mounted from `app.py` via a second `include_router` call. The existing `routes.py` (`/health`, `/execute`) is left as-is. Three alternatives were considered. **Append all five routes to the existing `routes.py`** was rejected because it would balloon a focused 40-line file to ~200 lines mixing stateless and stateful concerns, lose the per-resource APIRouter discipline, and obscure the OpenAPI grouping. **Split sessions across multiple files** (e.g. `sessions_crud.py` for create/list/get/delete + a separate file for execute) was rejected as premature splitting — five endpoints fit comfortably in one file, and per-verb splitting commits to a shape with no current justification. **The chosen one-file-per-resource-family pattern** is the FastAPI canonical layout and leaves room for future resources (jobs, metrics, etc.) to each take their own slot without disturbing siblings. OpenAPI auto-documentation groups by `tags=["sessions"]` so `/docs` displays all five together under a "sessions" heading.

### Substep 5: Per-session execute lock lives as an `asyncio.Lock` on `_Entry` + new `acquire_for_execute()` context manager on the registry

The substep-1 "reject concurrent execute with 409" decision required *somewhere* in the stack to own one `asyncio.Lock` per session. Three alternatives were considered. **A module-level `dict[sid, asyncio.Lock]` in `sessions.py`** was rejected because it requires route handlers to explicitly create/remove locks on POST and DELETE, and the background idle-eviction sweeper has no callback into route code — so evicted sessions would leak a Lock entry per session forever. **A `weakref`-keyed lock dict** was rejected as fragile (weakref behavior with Lock objects is hard to reason about; tying GC to functional behavior is a footgun). **The chosen design** adds a `lock: asyncio.Lock = field(default_factory=asyncio.Lock)` attribute to the private `_Entry` dataclass, plus a new `async with registry.acquire_for_execute(session_id) as runtime:` context manager that atomically (a) looks up the entry, (b) raises `SessionNotFound` if absent, (c) raises the new `SessionBusy` exception if `entry.lock.locked()`, (d) bumps `last_used` (same as `get_runtime`), and (e) enters `async with entry.lock`. Lock auto-cleans on delete/aclose/eviction with zero lifecycle tracking. `field(default_factory=asyncio.Lock)` is critical — a bare `default=asyncio.Lock()` would share one Lock across every `_Entry` ever created (Python evaluates default values once at class-definition time). The substep-4 lock "registry doesn't enforce concurrency" still holds: the registry provides the lock primitive, but the *policy decision* (return 409) is at the route layer. `SessionBusy` is a top-level Exception, sibling of `SessionNotFound`, not a child of `SessionRuntimeError` — exception_handlers map them via flat class equality.

### Substep 5: Session HTTP responses use new pydantic `BaseModel`s in `api/schemas.py`, converted from `SessionInfo` via `model_validate(info, from_attributes=True)`

`SessionResponse(BaseModel)` and `SessionListResponse(BaseModel)` are added to `api/schemas.py` alongside the existing `ExecuteRequest`/`ExecuteResponse` (which the session execute endpoint reuses unchanged). Route handlers convert the registry's `SessionInfo` dataclass to `SessionResponse` at the API boundary using `SessionResponse.model_validate(info, from_attributes=True)` — pydantic's "look up each declared field as an attribute of the source object" mode. Two alternatives were considered. **Return `SessionInfo` directly from routes and rely on FastAPI's stdlib-dataclass auto-serialization** was rejected because it mixes layers (execution-layer dataclass appears in HTTP `response_model` annotation), breaks the existing "all HTTP shapes are pydantic BaseModel in api/schemas.py" convention, and makes it harder to add HTTP-only fields later. **Rebuild `SessionInfo` as a pydantic.dataclasses dataclass** was rejected because it drags pydantic into the execution layer for no real gain and introduces a different validation model than the rest of `session_registry.py` uses. The chosen design pays a one-line conversion cost per route in exchange for clean layer separation and OpenAPI documentation with field descriptions. `SessionListResponse` wraps the list as `{"sessions": [...]}` rather than returning a bare top-level array — leaves room for future top-level fields like pagination or total count without breaking the response shape.

### Substep 5: Registry/runtime exceptions map to HTTP status via `@app.exception_handler` in `app.py`; `SessionTimeout` is the lone inline exception in the execute route

Four `@app.exception_handler(...)` decorators in `app.py` map: `SessionNotFound`→404 (`{"detail": "session not found"}`), `SessionBusy`→409 (`{"error": "session_busy"}` — the exact body shape locked in substep 1 decision 2), `SessionTerminated`→410 (`{"detail": "session is no longer running"}`), `SessionProtocolError`→500 (`{"detail": "internal protocol error"}` plus `logger.exception(...)` because protocol errors are server-side bugs we want stack-traced). `SessionTimeout` is *not* mapped via exception_handler — instead, the `execute_in_session` route catches it inline and returns `ExecuteResponse(timed_out=True, exit_code=-1)` at HTTP 200. Three sub-decisions drove this. **Why exception_handler not inline `try/except`?** Routes stay thin (three lines each, not seven); the mapping table is grep-able in one place; new exception types added later only need one new decorator, not edits to every route. **Why 410 for terminated, not 503 or 404?** 410 Gone is the canonical "resource was once available, now permanently gone" status; 503 implies "retry might help" (misleading — a dead session is dead); 404 implies "never existed" (also misleading). **Why 200 + timed_out=True for timeout?** Mirrors Phase 1's locked "timeout is data, not an exception" contract for stateless `/execute` — client code can use a single body-parsing path for both endpoints. Caller learns the session died on their *next* call (which returns 410). The exception_handler pattern is awkward for this case because the success-shaped response body (`ExecuteResponse`) is what we want to return; doing that from a global handler would lose the route's context, so we handle it where it's raised.

### Substep 5: Auth is applied router-level via `dependencies=[Depends(require_api_key)]` on the APIRouter, not per-route or app-wide

The `APIRouter(prefix="/sessions", tags=["sessions"], dependencies=[Depends(require_api_key)])` declaration applies the existing Phase 1 bearer-token check to every route on the router. Two alternatives were considered. **Per-route `dependencies=[Depends(require_api_key)]`** (same pattern as the current `/execute`) was rejected as DRY-violating — five copies of the same dependency, easy to omit on a future addition. **App-level dependency** (gate every endpoint) was rejected because it would also gate `/health`, which Phase 1 deliberately decided is a public liveness probe; reversing that was out of scope for substep 5. The router-level form is the FastAPI canonical pattern for "every route in this resource family needs the same gate." `require_api_key` is reused from `api/auth.py` exactly as-is — Phase 1's `secrets.compare_digest` + `HTTPBearer(auto_error=False)` design composes forward without modification. When `dev_api_key == ""` (the default), the check is a no-op pass-through, so tests that don't set the env var don't need any auth handling.

### Substep 6: Warm pool sized by `session_pool_size: int = Field(default=0, ge=0)` on Settings (opt-in)

A new Settings field, defaulting to 0 (pool disabled). Two alternatives were considered. **Default of 2 or 3 (always-on)** was rejected because §6.4 mandates pool *capability* but not always-on activation; default-on would silently spawn extra containers in every test, every dev environment, every learning user's startup — surprising new behavior for someone who didn't ask for it. **Hardcoded module constants** were rejected for the same reason substep-4 rejected hardcoded idle-timeout constants: tests would need monkey-patching rather than the clean `Settings(**overrides)` pattern. With `default=0`, every existing behavior is bit-identical to substep 5; opting in is a single env var (`KESTREL_SESSION_POOL_SIZE=N`). The validation is `ge=0` (greater-or-equal) rather than `gt=0` because 0 is a legitimate value — "pool disabled" — not an error.

### Substep 6: Pool state and logic live inline on `SessionRegistry`, not in a separate `SessionPool` class

Two new attributes (`_pool: list[SessionRuntime]`, `_refill_tasks: set[asyncio.Task]`) plus two private methods (`_schedule_refill`, `_refill_one`) are added directly to `SessionRegistry`. Two alternatives were considered. **A separate `SessionPool` class** in `src/kestrel/execution/session_pool.py` would have given single-responsibility separation. Rejected because: (a) pool state is tightly coupled to registry state — both manage `SessionRuntime` instances, both follow the same `start`/`aclose` lifecycle; (b) separating them creates a parallel lifecycle to coordinate (registry.start() must also pool.start(), two aclose() calls, etc.); (c) substep-3/4 precedent — premature abstraction is avoided until at least three callers/consumers justify it. **Dependency-injected `SessionPool`** was rejected for the same reasons plus DI overhead with no real test benefit (we can already disable via `session_pool_size=0`). The inline design adds ~30 lines to an existing file vs. ~80 lines for a new file with parallel state. If the pool grows complex logic (eager-vs-lazy modes, multi-tier pools, etc.), extraction is straightforward and bounded.

### Substep 6: Eager refill via `asyncio.create_task(self._refill_one())` scheduled per checkout, not a periodic background sweeper

Every `create()` that pops from the pool fires `self._schedule_refill()` once, which schedules one background `_refill_one()` task. Three alternatives were considered. **Lazy periodic background sweeper** (sibling of the idle-eviction sweeper, wakes every N seconds and tops up) was rejected because between sweeps the pool can stay depleted, defeating the latency win that justifies the pool's existence. **Hybrid eager + lazy safety net** was rejected as premature complexity — same "two mechanisms doing the same job" pattern substep 4 rejected for idle eviction. **The chosen eager design** keeps the pool aggressively at target size with minimal latency between depletion and refill. Refill tasks are tracked in `self._refill_tasks: set[Task]`; each task self-removes via `add_done_callback(self._refill_tasks.discard)`. The strong-reference set is necessary because asyncio holds tasks only via weak references — without anchoring them, garbage collection could destroy a task mid-spawn. `_refill_one()` uses the double-check pattern (`if self._closed`, `if len(_pool) >= target`) at both pre-spawn and post-spawn boundaries to handle races: another refill landing during our await, or `aclose()` being called while we were spawning. Pre-spawn checks are optimistic fast-bail; post-spawn checks are defensive — if the runtime we just spawned has nowhere to go, we close it ourselves rather than leak.

### Substep 6: The sweeper ignores pool entries — pool lives in `_pool`, not `_sessions`, so no special-case code needed

Pool entries are stored in `self._pool: list[SessionRuntime]`, separately from active sessions in `self._sessions: dict[str, _Entry]`. The idle-eviction sweeper (`_sweep_once`) iterates `self._sessions.items()` only — pool entries are invisible to it by construction. Two alternatives were considered. **A separate `session_pool_max_age_seconds` Setting** that ages out pool entries via the sweeper was rejected because pool runtimes are freshly-spawned containers with `--read-only` rootfs and `--tmpfs /tmp` — they accumulate basically nothing over time, and the hypothetical "pool entry got into a weird state" failure mode being protected against is extremely unlikely (no user code has run in them). **Treating pool entries the same as active sessions** (sweeper closes both, refill replenishes pool) was rejected because it would create thrash where the sweeper destroys warm containers right after refill creates them — defeating the warmth that justifies the pool. The chosen approach has zero new code (the structural separation between `_pool` and `_sessions` is what enforces the behavior). The test `test_sweeper_ignores_pool_entries` pins this by calling `_sweep_once(timeout_seconds=-1.0)` (which would evict every active session) and asserting `_pool` is untouched.

### Substep 6: `aclose()` awaits in-flight refills, then drains pool + active in one `asyncio.gather`; explicitly clears `_refill_tasks` after await

`aclose()` now does four steps: (1) cancel sweeper as before, (2) `await asyncio.gather(*pending_refills, return_exceptions=True)` so any spawn-in-progress completes before we drain, (3) `self._refill_tasks.clear()` for shutdown determinism, (4) `await asyncio.gather` over pool runtimes + active session runtimes concurrently. Three alternatives were considered. **Don't wait for in-flight refills** was rejected because a refill that completes after `aclose()` returns would have nowhere to put its spawned runtime — the orphan would survive until the next-startup orphan-sweep, which is a real but ugly fallback. **Drain pool first, then active sessions** sequentially was rejected because pool runtimes are idle (doing nothing); closing them first delays the close of active runtimes for no benefit. **The chosen design** is deterministic — by the time `aclose()` returns, every container the registry knew about (active and pool) is closed, and `_refill_tasks` is empty. The explicit `_refill_tasks.clear()` is load-bearing: `add_done_callback` schedules callbacks via `loop.call_soon`, so done-callbacks fire on a *subsequent* loop iteration after `gather` returns. Without the explicit clear, the set would still contain done-tasks for one tick. Tests that assert post-aclose registry state expect the clean shape; an early version without the explicit clear caused exactly that test to fail with done-tasks lingering in the set.

### Substep 7: Architecture model — Redis as a shared *directory* + sticky routing, not shared session state

Brief §6.4 mandates a "Redis-backed registry" for multi-worker scaling. The naive reading — move the `dict[session_id, _Entry]` into Redis so every worker shares it — is impossible, and the reason is the load-bearing insight of the whole substep: an `_Entry` holds a `SessionRuntime`, and a `SessionRuntime` is a live Docker container *plus the stdin/stdout pipes to it*. Those pipes are OS file descriptors owned by one specific process. Redis stores JSON, not file descriptors; a different uvicorn worker (different PID, possibly different host) cannot "load the runtime from Redis" and write to pipes that don't exist in its address space. Three architectures were considered. **Option A — Redis as a shared directory + sticky routing:** Redis holds only the serializable metadata (`session_id → {created_at, last_used, owner_worker_id}`); each worker keeps its own in-process map of the runtimes *it* owns; `GET /sessions` reads all of Redis for cross-worker visibility; execute requests must reach the owning worker (sticky load balancer). **Option B — detach the container comms:** replace `docker run -i` pipes with a unix socket on a shared volume so any process can reattach — rejected because it breaks the substep-1 protocol lock, needs a shared filesystem between workers (fails multi-host), and is a far larger change. **Option C — directory-only, minimal:** Redis just persists metadata for restart-reporting with no routing — rejected as doing and teaching the least while still not making sessions genuinely multi-worker. Chose **Option A**: it is the design real systems converge on precisely because of the file-descriptor constraint, and it keeps every prior Phase 4 lock intact — the substep-1 JSON-line-over-pipes protocol is untouched, and the substep-5 per-session `asyncio.Lock` and substep-6 warm pool both stay in-process per worker (see the two entries below).

### Substep 7: Pluggable-backend shape — `SessionRegistry` Protocol + `InMemorySessionRegistry` (renamed) + `RedisSessionRegistry`, selected by `Settings.session_backend`

The single concrete `SessionRegistry` class becomes a `runtime_checkable` `Protocol` (the interface), with two implementations: the existing in-memory class renamed to `InMemorySessionRegistry`, and a new `RedisSessionRegistry`. A `Settings.session_backend` Literal (`"memory"` | `"redis"`) plus a factory pick one at startup. Two alternatives were considered. **Keep the name `SessionRegistry` for the in-memory class** (add only `RedisSessionRegistry`) was rejected: it produces asymmetric naming where `SessionRegistry` sits beside `RedisSessionRegistry` and no longer signals "in-memory," which misleads a future reader. **An ABC base class with `@abstractmethod`** was rejected because it would introduce a second abstraction style into a repo that already established the `Protocol` pattern for swappable backends in Phase 2 (`Executor`). Choosing the Protocol form keeps Kestrel consistent: backends are structural, selected by a `Settings` Literal + factory, and routes/tests depend on the interface, never the concrete class. The rename ripples into `app.py` and the `tests/conftest.py` fixtures — accepted as a one-time mechanical cost in exchange for honest naming and pattern consistency.

### Substep 7: Redis key schema — hash per session + a `sessions` set index + a Redis TTL backstop

Each session is a Redis hash, `session:<id>`, with fields `created_at`, `last_used`, `owner_worker_id`. A `sessions` set holds every live id so `list()` is one `SMEMBERS` plus pipelined `HGETALL`s rather than a `SCAN` over the keyspace. Critically, **the app-level idle-eviction sweeper remains the real eviction mechanism** — it is the only thing that can actually kill a container — and on each execute the owning worker also refreshes a Redis `EXPIRE` on the hash, set generously longer than the idle timeout. Two alternatives were considered. **No Redis TTL, sweeper only** was rejected because it leaves one failure mode uncovered: if a worker crashes, its session hashes linger in Redis (and in the `sessions` set) until some worker's startup-reconciliation happens to remove them — `GET /sessions` would meanwhile list sessions that no longer exist. **Redis TTL as the sole expiry mechanism** was rejected outright: when Redis drops an expired key it cannot kill the container, so every idle eviction would orphan a container. The chosen hybrid gives the two mechanisms disjoint jobs — the sweeper kills live containers and cleans their keys; the TTL is a self-healing garbage collector for *directory entries whose owner worker died* — at a cost of one `EXPIRE` call per execute.

### Substep 7: Redis-down failure mode — fail hard with HTTP 503, no in-memory fallback

When the configured backend is `"redis"` and Redis is unreachable, `/sessions/*` routes return `503 Service Unavailable`. Two alternatives were considered. **Fall back to in-memory** was rejected because the degradation is silent and corrupting: cross-worker visibility vanishes without the caller knowing, two workers begin to disagree about what sessions exist, and when Redis returns the states have diverged with no reconciliation path — a silent inconsistency is strictly worse than a clean error. **A circuit breaker with retry** was rejected as Phase 6/7-grade resilience machinery (breaker state, retry policy, health probes) far beyond what Phase 4 needs. Failing hard tells the truth: the session directory *is* the source of truth, so when it is down, session operations are down. The blast radius is contained — `/execute` (stateless) and `/health` (public liveness) do not touch Redis and stay up.

### Substep 7: Cross-worker DELETE — remove the directory entry immediately; the owning worker's sweeper reconciles

`DELETE /sessions/{id}` for a session owned by a *different* worker (sticky routing usually sends it to the owner, but the design must answer the case where it does not): any worker may immediately `DEL session:<id>` and `SREM sessions <id>`. The owning worker's existing sweep loop gains a reconciliation pass — for each session in its local map, check whether it still exists in Redis, and if not, close that container. So the directory entry disappears at once and the container is reaped on the owner's next sweep tick (bounded by `session_sweep_interval_seconds`, default 60 s). Two alternatives were considered. **Redis pub/sub signalling** (publish `terminate <id>`; every worker runs a subscriber task that kills the container promptly) was rejected as premature: it adds a long-lived subscriber task and a new failure mode (missed messages) to every worker for an instant-teardown property nothing in Phase 4 requires. **Reject cross-worker DELETE with 409** was rejected as brittle — it outsources correctness to the load balancer, and the moment routing is not sticky (failover, misconfig, a direct call) a session becomes undeletable. The chosen approach reuses machinery that already exists (the sweep loop), needs no new transport or lifecycle, and the reconciliation loop — local worker state continuously converging to the Redis source of truth — is the pattern that makes Option A's architecture robust in general.

### Substep 7: `get_info` and `list` became `async` on the `SessionRegistry` Protocol; `get_runtime` stayed sync

Surfaced during implementation, not in the five design-lock entries above. When the single `SessionRegistry` class became a `Protocol` with two implementations, `get_info` and `list` had to change from sync to `async`. The in-memory backend's versions do no I/O and never need `await` — but the Redis backend's must round-trip to Redis, and a single Protocol cannot be sync for one backend and async for the other on the same method. So the interface takes the wider (`async`) shape: the in-memory `get_info`/`list` gained the `async` keyword with otherwise-unchanged bodies, and the two call sites in `api/sessions.py` gained `await`. `get_runtime` deliberately stayed **sync** — a `SessionRuntime` always lives in the worker that created it, so even the Redis backend's `get_runtime` only touches a local dict; no backend needs it to be async. The resulting asymmetry (`get_info`/`list` async, `get_runtime` sync) is principled: a method is async on the interface iff *some* backend genuinely needs to await inside it. Two alternatives were rejected: making `get_runtime` async too "for uniformity" (it would force `await` on a pure dict lookup at every call site for no backend's benefit) and keeping `get_info`/`list` sync with the Redis backend using a blocking sync client (blocking I/O on the event loop — a non-starter for an async service).

## Phase 5 Decisions (Rich Outputs)

### Substep 1: 5-scope — Rich outputs apply only to the session execute route

Rich outputs (matplotlib plots, pandas DataFrames, files) flow only through `POST /sessions/{id}/execute`. The stateless `POST /execute` route keeps the Phase 1 contract — `stdout`, `stderr`, `exit_code`, `timed_out`, nothing more. Two alternatives were considered. **Both paths via a unified kernel** (wrap stateless calls as ephemeral kernel sessions) was rejected because it re-opens Phase 2's "fresh container per request, no warm pool" lock for the stateless path: either every stateless call pays the kernel's ~400 ms cold-start, or a stateless warm pool — explicitly rejected in Phase 2 to prevent cross-request leakage — gets reintroduced. It also drifts a stable public contract for callers who never asked for rich outputs. **Both paths via two mechanisms** (kernel hooks for sessions, a wrapper script for stateless) was rejected because the recurring cost of keeping two implementations behaviorally identical across matplotlib/pandas version bumps and MIME edge cases is exactly the kind of lock-vs-relitigate cost Phase 4's design-lock pattern was designed to avoid. The Phase 4 kernel was already the natural intercept point for stateful execution; the Brief frames the whole feature as "Jupyter kernel experience" (§1) which is fundamentally session-shaped. The cost is one explicit footnote in the API docs: rich outputs are session-only, and a one-shot rich-output user composes three calls (create → execute → delete) instead of one — accepted as cheap composition.

### Substep 1: 5-shape — Each rich output is a discriminated-union pydantic model on `type`

The `outputs` array carries entries of a `RichOutput` discriminated union — `PlotOutput | DataFrameOutput | FileOutput`, joined with `Annotated[..., Field(discriminator="type")]`. Each member is a small pydantic model that owns its own metadata: `PlotOutput` locks `mime_type` to `image/png` and carries base64-encoded PNG bytes; `DataFrameOutput` carries `mime_type="application/json"`, the `to_dict(orient="split")` payload, and a `shape: tuple[int, int]` (rows, cols); `FileOutput` carries a variable `mime_type`, a `filename`, and base64-encoded bytes. Two alternatives were considered. **Minimal `{type, data}` (literal Brief §4.2 example)** was rejected because MIME-type handling is an explicit Phase 5 deliverable and has no home in a two-field shape — it would either go into `data` as an untyped dict key (defeats the schema) or get cargo-culted into top-level later. Per-type extras like `filename` have the same problem. **Standard flat `{type, mime_type, data}`** was rejected because it lets MIME live cleanly but leaves per-type extras homeless; the schema cannot express "FileOutput has filename, PlotOutput does not." The chosen discriminated union matches the project's heavy-pydantic house style, makes the OpenAPI schema self-documenting (which satisfies the "Output format documentation" Phase 5 deliverable for free), and costs ~30–40 lines in `api/schemas.py` upfront in exchange for never having to reshape the response model when a fourth output type appears.

### Substep 1: 5-model — `SessionExecuteResponse(ExecuteResponse)` carries `outputs`; stateless `ExecuteResponse` unchanged

A new pydantic model `SessionExecuteResponse` extends `ExecuteResponse` and adds `outputs: list[RichOutput] = Field(default_factory=list)`. The session execute route's return type bumps from `ExecuteResponse` to `SessionExecuteResponse`; the stateless `POST /execute` keeps `ExecuteResponse` untouched. Two alternatives were considered. **Add `outputs` to `ExecuteResponse` directly with default `[]`** was rejected because the stateless route's OpenAPI schema would then advertise an `outputs` field that the route can never populate — schema dishonesty that contradicts the 5-scope lock above and forces the API docs into the awkward "this field exists on stateless but is always empty" footnote. **A flat re-declared `SessionExecuteResponse(BaseModel)` with no inheritance** was rejected because it requires copy-pasting every `ExecuteResponse` field and re-duplicating any future shared additions; the IS-A relationship between the two responses is real (a session execute response truly is an execute response with extras), so inheritance models it correctly. The chosen subclass adds ~5 lines, preserves the 20 parametrized `test_execute.py` tests bit-identically, and FastAPI's openapi generator picks up the per-route schema split automatically. The 5-status lock from Phase 4 substep 5 (returning `ExecuteResponse(timed_out=True, exit_code=-1)` inline on `SessionTimeout`) is preserved by returning a `SessionExecuteResponse(timed_out=True, exit_code=-1, outputs=[])` instead — same shape, satisfies the new return type.

### Substep 1: 5-capture — Hybrid capture inside the kernel (implicit for plots + DataFrames, explicit for files)

The kernel uses three different capture mechanisms, one per output type, chosen to match how Jupyter handles each. **Matplotlib** is captured implicitly: `matplotlib.use("Agg")` is forced at kernel boot (before any user code can `import matplotlib`); after each execute the kernel walks `plt.get_fignums()`, calls `figure.savefig(buffer, format="png")` on each open figure, base64-encodes the bytes into a `PlotOutput`, then `plt.close(fig)` to clear the registry for the next execute. **DataFrames** are captured implicitly via the last-expression rule: the kernel `ast.parse`s the incoming code, checks whether `body[-1]` is an `ast.Expr`, and if so splits the code — runs everything-but-last via `exec`, evaluates the last expression via `eval`, and if the resulting value is a `pandas.DataFrame` dispatches to a `DataFrameOutput`. This is exactly the IPython/Jupyter convention. **Files** are captured explicitly via a watched-directory convention: the kernel pre-creates `/workspace/outputs/` (a new tmpfs mount, separate from `/tmp` scratch) at boot; after each execute it scans only that directory for new or modified files since the previous execute, slurps each into a `FileOutput`, then clears the directory. The user opts in by writing to that path (`df.to_csv("/workspace/outputs/report.csv")`). Two alternatives were considered. **Fully implicit (Jupyter-style for everything including files)** was rejected because scanning all of `/tmp` would capture incidental writes — Python bytecode caches, library temp files, user scratch — leading to bloated payloads and potential leakage. Even a sub-path convention is "implicit" by name only; files genuinely need explicit opt-in because there's no Jupyter-style "natural expression" for them. **Fully explicit (user calls `kestrel.publish_plot()` / `publish_dataframe()` / `publish_file()`)** was rejected because it breaks the Brief's "Jupyter kernel experience" framing — Jupyter users expect plots and DataFrame literals to appear without ceremony. The hybrid matches each output type's natural pattern, eliminates the file-scanning footgun, and is the closest reading of the Brief's mixed phrasing ("Matplotlib plot capture / Pandas DataFrame detection / **File output support (write inside container, return via API)**" — the file phrasing already implies an explicit user write). The kernel's JSON-line reply protocol locked in Phase 4 substep 1 gains an `outputs: list = []` field as a backward-compatible amendment; a missing `outputs` key parses as empty, so kernel versions without the field still work. The runtime image bumps from `kestrel-runtime:0.3.0` to `:0.4.0` with matplotlib + pandas added (~130 MB combined) and the `/workspace/outputs/` tmpfs mount declared in `SessionRuntime.start()`'s docker run flags.

### Substep 1: 5-limits — Tight env-var-configurable byte caps; excess outputs are dropped whole and surfaced in a separate `dropped_outputs` field

Per-output byte caps default to 2 MB for plots, 1 MB for DataFrames, 5 MB for files; per-execute total cap is 10 MB; file count cap is 10 per execute. All five values are exposed as new Settings fields (`KESTREL_RICH_OUTPUT_PLOT_MAX_BYTES`, `_DATAFRAME_MAX_BYTES`, `_FILE_MAX_BYTES`, `_TOTAL_MAX_BYTES`, `_FILE_MAX_COUNT`). When an output exceeds its cap, the kernel/host **drops it whole** — there is no mid-payload truncation, because a half-encoded base64 PNG is a broken image and a truncated DataFrame JSON is invalid JSON, so the stdout-style "preserve beginning + `…truncated…` marker" pattern does not transfer. Each drop appears in a new `dropped_outputs: list[DroppedOutput]` field on `SessionExecuteResponse`, where `DroppedOutput` is `{type, reason, size_bytes, filename?}` with `reason` a Literal of `per_output_cap | total_cap | file_count_cap`. Two alternatives were considered for the cap values. **Generous Brief-verbatim caps** (no per-plot/per-DataFrame caps, 5 MB per file, 50 MB total) was rejected because it breaks Phase 1's house style of choosing defaults tighter than the Brief's spec ceiling (KESTREL_EXECUTE_OUTPUT_CAP_BYTES = 64 KiB per stream, not the Brief's 10 MB) — relaxing later via an env var is cheap; tightening after users hit problems is not. **Middle-ground caps** (5/2/5 MB per output, 20 MB total) were rejected for being middling for the sake of middling, matching neither the Brief nor the Phase 1 precedent. Two alternatives were considered for the drop signal. **An in-band stderr line** (`[kestrel] plot 2 (12.3 MB) dropped`) was rejected because it pollutes the user's own stderr with system messages, breaking the principle that stderr is user code's output. **A sentinel `TruncatedOutput` type in the `RichOutput` union** was rejected because it mixes data and meta in the same list (clients handling `RichOutput` would need to branch on every entry) and re-opens the 5-shape lock. The chosen separate `dropped_outputs` field keeps `outputs` clean (data only), keeps stderr sacred, and self-documents in the OpenAPI schema. **Where the cap logic runs** (kernel-side vs host-side) was deliberately left out of this lock as a 1-day-reversible implementation choice; substep 2's note: lean host-side so the kernel stays "capture + emit raw" and Settings live in one place — reconsider only if pipe bandwidth becomes a real issue.

### Substep 7: 7-total-ordering — Total-cap budget is spent in capture order; first-emitted wins

The substep 1 lock 5-limits specified the 10 MiB per-execute total cap but did not specify which outputs survive when the budget is tight. Substep 7's implementation makes the choice: outputs are processed in the kernel's emission order (plots first, then the AST-last-expression dataframe, then files in sorted filename order), and the first to arrive get the budget. Two alternatives were considered. **Prioritise by type** (e.g. always include plots before files regardless of order) was rejected because it requires communicating a hidden priority order to users — "your dataframe was dropped even though file X was kept; the reason isn't visible in the code." **Prioritise by size** (smallest-first to maximise count) was rejected for the same reason plus a perverse-incentive one: users could game inclusion by emitting tiny dummy outputs. The chosen capture-order rule has the simplest mental model — "the order you emitted things is the order the budget was spent" — and aligns with the substep 6 lock that file ordering is `sorted(os.listdir())` (deterministic, user-visible). The cost is that a user who wants a specific output guaranteed must emit it first; this is documented in the API docs. Per-output drops do *not* consume total budget (enforced by check ordering: `if per_output_cap` → `elif total_cap` → `else accept` plus `total_bytes += size_bytes` living inside the accept branch); test `test_per_output_cap_drops_do_not_consume_total_budget` proves it.

### Substep 7: 7-redis-kwargs-fix — Pre-existing Redis-registry kwargs gap fixed as a separate commit before substep 7 feature work

Pre-flight review for substep 7 surfaced that `redis_session_registry.py` was not passing any of the `rich_output_*` kwargs to `SessionRuntime.start()` at either of its two call sites (the create-fallback at line 119 and the pool-refill at line 365). The in-memory registry passed them; the Redis registry didn't — silently falling back to `SessionRuntime`'s hard-coded defaults and ignoring Settings values on the Redis backend. Introduced when substeps 4-6 added the kwargs to the in-memory registry without mirroring the change on the Redis side; not caught by tests because cap behaviour is exercised at the `SessionRuntime` layer, not via a Redis end-to-end with custom caps. Two options for fixing: **fold into substep 7** (one commit covers both the bug and the feature) was rejected because git history would attribute the fix to feature work, hiding the pre-existing nature of the bug. **Separate fix commit first** was chosen — commit `dbe94c6` cleanly attributes the fix to its origin (substeps 4-6) and lets the substep 7 commit be purely about the new `total_max_bytes` plumbing and the response-model bump. No test was added specifically for the Redis-fix because the existing in-memory tests cover the cap-application logic; the fix is config-plumbing parity, not new behaviour. Future kwargs additions to `SessionRuntime.start()` should be mirrored in both registries — flagged as a maintenance footgun for future maintainers.

## Phase 6 Decisions (Streaming)

### Substep 1: 6-scope — Streaming is a session-only feature; stateless `POST /execute` stays sync

WebSocket streaming is exposed only on the session route as `WebSocket /sessions/{id}/execute/stream`. The stateless `POST /execute` keeps the Phase 1 contract — single request, single JSON response, no streaming variant. Two alternatives were considered. **Stateless-only** matches Brief §4.1's literal endpoint name `/execute/stream`, but it puts the streaming experience on the route where long-running code is least natural (stateless executes are by Phase 1 lock killed at the 5s timeout, so the streaming window is tiny; the Brief's framing of Kestrel as "Jupyter kernel experience" §1 places long-running work in sessions). **Both routes** doubles the implementation surface for marginal benefit — the same one-shot user who wants streaming output can create→stream→delete in three calls, mirroring Phase 5's one-shot-rich-outputs pattern. Phase 5's 5-scope lock established the precedent "richer experiences live on sessions, stateless stays the simple contract"; breaking that for Phase 6 would require explaining why streaming is categorically different from rich outputs (it isn't — both are session-shaped features in spirit). The Brief's endpoint name `/execute/stream` is taken as illustrative, not prescriptive: the actual path becomes `/sessions/{id}/execute/stream`. One footnote in the API docs documents the composition pattern for stateless users.

### Substep 1: 6-transport — WebSocket only; not SSE, not both

The streaming primitive is WebSocket (FastAPI's `@app.websocket(...)` route). Two alternatives were considered. **Server-Sent Events** is technically the better tool for one-way server→client streaming — simpler protocol (plain HTTP text/event-stream), built-in browser auto-reconnect via `EventSource`, better traversal of corporate HTTP proxies that block WebSocket upgrades. But the Brief is explicit: §1.1 lists "Streaming execution — real-time stdout streaming" and §4.1 names `WebSocket for streaming execution` as the endpoint mechanism; §2.1 component diagram labels client↔API as "HTTP/JSON, WebSocket". Deviating from a clear Brief mandate needs strong cause, and "SSE is simpler" doesn't clear that bar. WebSocket also gives the bidirectional channel free, opening room for future cancel-mid-execute, send-stdin, or "interrupt the kernel" features without re-locking the transport. **Both (WebSocket primary + SSE fallback)** was rejected as 2× transport-layer code for a problem the polling fallback (6-polling) is already meant to solve. FastAPI WebSocket support is excellent and the asyncio integration is native. Authentication via the same bearer token Phase 4 substep 5 wires for the session routes: WebSocket query-param or header check at the upgrade handshake.

### Substep 1: 6-polling — Polling fallback in Phase 6 but isolated to the final substep so it can be cut cleanly under scope pressure

Brief lists "Fallback to polling if WebSocket fails" as a Phase 6 deliverable. Polling adds substantial new state to Kestrel that doesn't exist anywhere else: a server-side per-execution output buffer (so reconnecting clients can replay missed chunks), a cursor/offset mechanism so the client tracks what it's consumed, and either a polling endpoint (`GET /executions/{id}/output?since=<cursor>`) or a long-poll variant. Implementation cost is comparable to a full substep's worth of work. Three options were considered. **Defer to Phase 8 Polish & Ship** would honor the Brief eventually but breaks Phase 6's promise; documentation-as-fallback isn't a real fallback. **Bundle polling into the WebSocket substeps** risks scope creep without a clean abort point. **Isolate polling as the final pre-closeout substep** keeps the Brief deliverable in Phase 6 while letting us cut it to the next phase if implementation reveals unexpected complexity (e.g. the output-buffer state needs Redis backing for multi-worker sticky routing, which would re-open Phase 4 substep 7 territory). The substep plan reserves slot 6 for polling specifically so this carve-out is structurally explicit. If A/B telemetry (post-ship) shows the WebSocket route has near-100% success rate across real clients, polling can be reduced to documentation; if not, the substep ships as planned.

### Substep 1: 6-protocol — Typed JSON message discriminated union over WebSocket text frames

Every WebSocket message is a JSON object with a discriminator field `type` and message-specific fields. Pydantic models for each variant join via `Annotated[Union[...], Field(discriminator="type")]` — same pattern as Phase 5 substep 3's `RichOutput`. The variants: `{"type": "stdout", "data": "..."}` (one frame per line, UTF-8 decoded), `{"type": "stderr", "data": "..."}` (same shape, separate channel), `{"type": "heartbeat", "elapsed_ms": N}` (liveness; see 6-progress), `{"type": "result", ...full SessionExecuteResponse}` (final frame, identical shape to Phase 5 substep 7's `SessionExecuteResponse` so rich outputs flow over streaming with zero shape changes — `outputs`, `dropped_outputs`, `exit_code`, `duration_ms`, `timed_out` all carried), `{"type": "error", "code": "...", "detail": "..."}` (server-side failure during streaming). Two alternatives were considered. **Raw text frames with sentinel markers** (e.g. prefix bytes for stream type) is more compact but loses type info, requires custom parsing on both ends, and breaks FastAPI's ability to auto-generate OpenAPI documentation. **Newline-delimited JSON inside text frames** mixes layers (framing in WebSocket + framing inside the frame). The chosen discriminated-union approach matches the project's pydantic-everywhere house style (Phase 5 5-shape, Phase 4 5-shape), generates self-documenting OpenAPI schema for clients, and lets the final `"result"` message reuse the existing response model verbatim. WebSocket close codes carry the final session state for session-level errors (1000 normal, 1011 server error for protocol failures, 4404/4409/4410 application-level mirrors of the HTTP exception map from Phase 4 substep 5 5-status).

### Substep 1: 6-backpressure — Natural back-pressure to the kernel, with a 30s send-stall safety disconnect

When the WebSocket client consumes slower than the kernel produces, the host-side streaming runtime pauses reading from kernel stdout (`asyncio.StreamReader` natural back-pressure) which eventually fills the kernel's own stdout pipe buffer, which blocks the kernel's `print` calls — back-pressure all the way to user code. Combined with a 30-second cap on consecutive back-pressured send-attempts: if the WebSocket send still can't drain after 30s, the streaming runtime closes the WebSocket (close code 1011) and treats the disconnect per 6-disconnect (kill the execute, release the session lock). Three alternatives were considered. **Drop messages silently** (keep reading kernel, discard chunks client can't consume) was rejected because lossy streaming for a code-execution service is misleading — a user trying to debug long output would never know they missed lines. **Disconnect immediately on any back-pressure** (e.g. 1s timeout) was rejected as too aggressive — bursts of output (a large `print(df)`) momentarily fill the WebSocket buffer even on fast clients; 1s would false-positive on normal traffic. **Pure back-pressure with no upper bound** was rejected because a permanently-slow client could indefinitely stall a kernel that's still consuming session resources (memory, container slot in the pool). The chosen approach preserves correctness (no silent drops), follows asyncio's idiomatic back-pressure mechanism, and bounds the worst case via the 30s safety disconnect. The 30s value lives in `Settings.stream_backpressure_timeout_seconds` (default `30.0`); the per-execute `execute_timeout_seconds` cap (currently 5s default for stateless, longer-tolerant for sessions) still applies independently.

### Substep 1: 6-disconnect — Client disconnect immediately kills the running execute and releases the session lock

When the WebSocket connection closes mid-execute (client-initiated disconnect, network drop, the 6-backpressure safety timeout, or any error), the streaming runtime kills the docker container via the existing `_docker_kill` helper, marks the SessionRuntime as `_terminated=True`, and releases the per-session execute lock so the next request on the session can proceed. The session itself is gone (per Phase 4 substep 3's "terminate on any failure" semantics — the kernel can't be revived after kill). Two alternatives were considered. **Let the execute finish; discard output** keeps the kernel running for nobody — the execute-lock stays held by a phantom owner, the session becomes unusable until the kernel exits naturally (potentially hours), and resources stay consumed. **Buffer output for late reconnect** is meaningful only with polling fallback (6-polling) and requires the same server-side output-buffer machinery; it's left as a possible enhancement once polling ships, but the streaming path itself does not buffer. The chosen behaviour mirrors HTTP request semantics: client disconnects = request cancelled. Critical caveat: when the streaming WebSocket is killed, the *session itself* terminates (because the kernel was killed). Subsequent execute requests on the same session_id get HTTP 410 Gone via the Phase 4 substep 5 5-status mapping. This is a stronger consequence than the equivalent HTTP-disconnect for the non-streaming session route (which leaves the session alive); accepted because the streaming kernel needs to be killed to release output-pipe back-pressure, and killing the kernel terminates the session by Phase 4 design.

### Substep 1: 6-progress — Application-layer heartbeat every 5 seconds; configurable, distinct from WebSocket transport pings

The streaming runtime emits `{"type": "heartbeat", "elapsed_ms": N}` every 5 seconds if no other message has been sent in that interval. Clients can use these to distinguish "execute is silently working" from "execute is frozen / connection is dead." Three alternatives were considered. **Periodic stats** (`{"type": "stats", "stdout_bytes": ..., "elapsed_ms": ...}`) was rejected as more surface for unclear benefit — clients can count their own received bytes; the only useful new datum (elapsed_ms) is already in heartbeat. **Skip the application-layer heartbeat; rely on WebSocket ping/pong** was rejected because ping/pong is transport-layer (RFC 6455 §5.5.2-3) and most browser/library APIs hide it from application code — the heartbeat needs to be observable to client logic. **Defer to user code via a kernel-side `kestrel.progress(...)` helper** was rejected because it requires baking a helper into the kernel runtime, adds API surface, and doesn't solve the "is anything happening at all" case (user code that doesn't call the helper looks frozen). The chosen 5-second interval is the same cadence as `SESSION_SWEEP_INTERVAL_SECONDS` default — feels frequent enough to be useful (UI spinner update cadence) without flooding the channel. Configurable via `Settings.stream_heartbeat_seconds` (default `5.0`); set to `0.0` to disable. Heartbeats reset the back-pressure timeout from 6-backpressure (a heartbeat is a normal send; if it can't drain in 30s, the same safety disconnect fires).

### Substep 1: 6-kernel — Backward-compatible kernel protocol extension: multiple JSON lines per execute, terminated by a `result` line. Image bumps to `kestrel-runtime:0.5.0`

The kernel's `_execute` is restructured to emit JSON lines progressively rather than buffering all stdout/stderr to the end. New line types `{"id": ..., "type": "stdout_chunk", "data": "..."}` and `{"type": "stderr_chunk", "data": "..."}` are emitted as the kernel reads its captured stdout/stderr buffers (which now drain incrementally on each `print` flush, via a custom file-like object that intercepts writes and emits chunks). The execute terminates with `{"id": ..., "type": "result", "exit_code": ..., "outputs": [...], ...}` carrying the final summary (rich outputs, exit code, duration). Three alternatives were considered. **Major kernel rewrite** (replace the buffer with a streaming producer top-to-bottom) is the cleanest design but risks the protocol-stability promise of Phase 4 substep 1 — the existing single-line "result-only" reply is what every non-streaming consumer expects. **New separate kernel mode** (keep `kernel.py` as-is; new `kernel_stream.py` for streaming sessions) doubles the kernel surface and needs sessions to declare their mode at start, which leaks the streaming-vs-non-streaming distinction into session lifecycle (currently sessions are uniform). The chosen **backward-compatible extension** keeps the single-line reply as a degenerate case (zero chunks, then `"result"`) — the existing non-streaming `SessionRuntime.execute()` path needs an update to consume the new multi-line shape: read lines until a `"result"` type arrives, ignore any `"stdout_chunk"`/`"stderr_chunk"` lines that arrive along the way (or coalesce them into the final response's stdout/stderr fields — cleaner for the non-streaming consumer). The kernel-side change is a custom `_StreamingStdout(io.TextIOBase)` (or similar) that intercepts `write()` to emit a chunk line and also buffers locally for the final result message's `stdout` field; same for stderr. Image tag bumps `:0.4.0` → `:0.5.0`; `:0.4.0` retained alongside for rollback (matches the `:0.3.0` retention pattern from Phase 5 substep 2). `Settings.executor_docker_image` default updates to `:0.5.0`.

### Substep 6: 6.6-mech — Long-poll only with `?wait=N` query param; `wait=0` degrades to short-poll

The polling fallback uses long-poll semantics by default: the client passes `?wait=<seconds>` on the read endpoint, and the server holds the request until new chunks arrive OR the wait timeout fires. `wait=0` returns immediately with whatever chunks are currently buffered (short-poll behavior on the same endpoint). Two alternatives were considered. **Short-poll only** (client polls on a fixed interval, server always returns immediately) was rejected because the request count and end-to-end latency are both higher than necessary — in the WebSocket-failure environments polling is designed for (corporate proxies blocking the upgrade), the HTTP path itself is usually fine, and a long-poll just looks like a slow request. **Separate endpoints per mode** was rejected as redundant — one endpoint with a `wait` parameter expresses both behaviours with no duplication. Server-side cost is one `asyncio.Event` per execution that pending GETs `wait_for` on; the event is set whenever a new chunk is appended to the buffer. Long-poll wait clamp lives in the same `Settings.stream_backpressure_timeout_seconds` family (or its own field — to be decided during implementation; substep 6 defaults to a hardcoded sensible value if no clear Settings home emerges).

**Implementation resolution (2026-05-22).** The wait clamp got its own Settings field — `polling_max_wait_seconds`, default 30.0 — rather than reusing `stream_backpressure_timeout_seconds`. The two govern unrelated things: the back-pressure timeout bounds how long a single WebSocket *send* may stall before the kernel is killed; the poll clamp bounds how long a GET may *hold* before returning empty. Coupling them onto one field would let an operator tuning one silently shift the other. A dedicated field also matches Kestrel's configure-every-operational-knob convention. Separately, the per-execution wakeup primitive is an `asyncio.Condition`, not the bare `asyncio.Event` this lock's text speculated: `Condition.wait_for(predicate)` re-checks the predicate under a lock — lost-wakeup-free and correct for multiple concurrent GETs on one execution, where a bare Event needs fragile set/clear ordering.

### Substep 6: 6.6-shape — Two HTTP endpoints: POST to start, GET to read

The polling lifecycle is split across two endpoints: `POST /sessions/{id}/execute/polling` accepts an execute request and returns `{execution_id}` immediately (fire-and-forget; the execute runs asynchronously in the background); `GET /sessions/{id}/executions/{execution_id}?since=<cursor>&wait=<seconds>` returns the next batch of chunks, the next cursor, and a `done` flag. Two alternatives were considered. **Combined start-and-first-read** (POST returns the first batch + cursor) would conflate start semantics with read semantics — once you've started, every subsequent call is a read; the POST is logically distinct. **Polling-as-SSE** was rejected because the Brief explicitly says "polling" (not "SSE"), and SSE is a new transport surface that the existing WebSocket route was supposed to be the singular streaming-transport solution for. Two endpoints, clean REST semantics, each does one thing.

### Substep 6: 6.6-cursor — Integer monotonic counter starting at 0

The server holds an append-only `list[StreamMessage]` per execution; the cursor is the next-to-read index. Client passes `?since=N` to get messages with index `>= N`. Two alternatives were considered. **Byte offset** doesn't fit — the protocol is structured messages, not a byte stream; converting between message-index and byte-offset would require both ends to agree on a serialization that doesn't naturally have one. **UUID per chunk** is overkill — UUIDs don't give ordering for free, so the server would still need an index for replay semantics; UUIDs would also balloon response size. The chosen integer cursor matches every other Kestrel pattern that needs sequence semantics (per-execute msg_id correlation, sweep-by-last-used timestamps, etc.) and is trivially encodable in URLs.

### Substep 6: 6.6-buffer — Buffer lives on `SessionRegistry`, scoped by `session_id`

The per-execution polling buffer (`_polling_buffers: dict[session_id, dict[execution_id, PollingBuffer]]`) attaches to the `SessionRegistry` rather than to the `SessionRuntime` or a new top-level component. Two alternatives were considered. **On `SessionRuntime`** would couple the buffer's lifetime to the runtime's, but polling buffers can outlive the active execute (the client may still be polling after the kernel has finished and the runtime is idle); that decoupling is awkward inside the runtime. **A separate `PollingBufferRegistry`** would require its own lifecycle wiring (`start()`/`aclose()`, sweeper, lookup) just to host one stateful component; premature abstraction. The chosen home — co-located with the lifecycle owner — gives three things for free: session delete → buffers auto-cleaned via the dict-of-dicts removal; idle session sweep → buffers go with the evicted session; and the existing sweep mechanism extends naturally to also evict expired buffers (see 6.6-evict).

### Substep 6: 6.6-sticky — Session-id-sticky routing handles multi-worker buffer placement; no Redis-backing needed

The polling URL `/sessions/{session_id}/executions/{execution_id}` includes `session_id` in the path. Phase 4 substep 7's multi-worker deployment already requires session-id-sticky routing at the load balancer (so that the worker holding the `SessionRuntime` receives the execute requests). The same sticky routing puts both the polling POST and all subsequent GETs on the same worker — the worker that has the buffer. The buffer therefore stays in-process per-worker, with zero serialization or Redis round-trips. Two alternatives were considered. **Redis-back the buffer** would re-open Phase 4 substep 7 territory: pickle/JSON-serialization of every `StreamMessage`, eviction races between workers, per-buffer TTL backstops in Redis — substantial new work the substep-1 6-polling lock explicitly called out as the cut-trigger. **Don't support multi-worker for polling** would split the multi-worker story (Redis backend supports sessions + WebSocket streaming but not polling) — confusing and a regression from the Phase 4 multi-worker promise. **This decision is what allows substep 6 to ship rather than be cut to documentation** — the sticky-routing insight makes the buffer placement clean.

### Substep 6: 6.6-evict — TTL after execute completion, default 60s

The polling buffer is evicted on a TTL clock that starts when the execute completes (success / timeout / error). Default TTL = 60 seconds, exposed as `Settings.polling_buffer_ttl_seconds`. The existing `SessionRegistry` sweep extends to also scan `_polling_buffers` and drop any with `completed_at + ttl < now`. Two alternatives were considered. **TTL after last read** (buffer evicts 60s after the last GET, regardless of execute state) was rejected because a slow execute that takes minutes could be evicted mid-stream if the client doesn't poll often enough — converts a wait-cycle into a buffer-eviction. **Hard message-count or byte cap** (drop oldest chunks when buffer exceeds N) is unnecessary because the per-execute total-cap from Phase 5 substep 7 (default 10 MiB) already bounds the buffer's worst-case size during execution; the cap that controls user-emitted output already controls the polling buffer for free. The chosen TTL-after-completion has one simple invariant: the buffer is alive throughout the execute, plus a 60-second grace window for late readers, then it's gone.

### Substep 6: 6.6-wrap — Wrap `execute_stream()` in a polling-specific orchestrator (don't add a sibling method)

The polling routes don't call `SessionRuntime.execute_stream()` directly. Instead a polling-specific orchestrator function (`_run_polling_execute(session_id, execution_id, code, registry)` in `sessions_polling.py`) is spawned as an `asyncio.create_task` from the POST handler. The orchestrator acquires the session lock via `registry.acquire_for_execute`, iterates `runtime.execute_stream(code)`, appends each message to the buffer, and sets the buffer's `new_chunk_event` so any waiting long-poll GET wakes up. Two alternatives were considered. **A new method on `SessionRuntime`**: `execute_polling(code, buffer)` that does the iteration internally was rejected because it duplicates the loop and conflates two responsibilities (the runtime should produce messages; deciding where they go is the layer above). **Refactoring `execute_stream` to optionally accept a buffer parameter** was rejected for the same reason plus an awkward type signature. The chosen wrap pattern reuses `execute_stream` unchanged and keeps the polling-specific orchestration in the polling-specific file — same decoupling precedent as Phase 4 substep 5 (route layer owns 409 policy; registry layer just provides the lock primitive).

### Substep 6: 6.6-loc — New file `src/kestrel/api/sessions_polling.py`

The polling routes live in a sibling file to `sessions_stream.py`. Two alternatives were considered. **Append to `sessions_stream.py`** would co-locate "all streaming-adjacent code" but mixes WebSocket lifecycle with HTTP-async-execute lifecycle — different routes, different state, different mental models. **Append to `sessions.py`** would put all HTTP routes in one place but mixes substep-5's synchronous-execute routes with substep-6's async-execute pattern (POST returns a handle; GET polls). The chosen new file follows the same precedent as substep 3's split of `sessions_stream.py` from `sessions.py`: different lifecycle, different concerns, file will grow. New router mounted in `app.py`: `app.include_router(sessions_polling_router)`. The polling buffer's home (`SessionRegistry._polling_buffers`) is the only thing this file shares with the rest of the codebase — clean seam.

## Phase 7 Decisions (Observability & Management)

### Phase 7 Scope (7-scope)

Phase 7 implements Brief §6.7: Prometheus metrics, request-ID tracing, PostgreSQL audit log, API-key management CLI, per-key rate limits, admin endpoints. The capability set divides into seven substeps, mirroring Phase 6's rhythm of three foundation substeps + two integrations + one polish + one closeout:

1. **Foundations — Prometheus metrics + request-ID propagation.** Add `prometheus_client`, mount `GET /metrics`, seed a small load-bearing metric set (HTTP counters/histograms, execution counters, session/stream/polling gauges). Extend the existing `X-Request-ID` middleware so `request_id` rides through session execute → kernel JSON-line `request_id` field → audit sink → stream/polling terminal frames.
2. **Postgres audit log — schema + async writer.** Introduce a `AuditSink` protocol with `NullAuditSink` + `PostgresAuditSink` implementations, chosen by `build_audit_sink(settings)` mirroring `build_session_registry`. One `audit_events` table; writes async-queued (see 7-audit-sync).
3. **API-key store — model + lookup.** Replace single `KESTREL_DEV_API_KEY` env var with a Postgres-backed key store. `require_api_key` switches to hash-lookup-verify. Dev-mode env var retained as boot-friendly shim that logs a loud warning.
4. **Key-management CLI.** `kestrel-keys` console-script entrypoint: `create <label>`, `list`, `revoke <id>`. Plaintext key printed once on create. Argparse-based (no new deps).
5. **Per-key rate limits.** Token-bucket per `(api_key_id, route_class)`. Storage follows `KESTREL_SESSION_BACKEND` (memory or Redis). 429 + `Retry-After` on exhaustion.
6. **Admin endpoints.** `GET/POST/DELETE /admin/keys`, `GET /admin/sessions`, `GET /admin/audit`. Gated by `admin` scope on the API key (not a separate key family).
7. **Closeout.** Documentation updates, README + ROADMAP + tag `v0.7.0`.

Stateless `POST /execute` keeps the Phase 1 contract; Phase 7 only adds (a) audit-log writes, (b) `/metrics` scraping, and (c) rate-limit enforcement gates on top — no response-shape changes to the existing endpoints.

### 7-pg-required — PostgreSQL is a hard Phase-7 dependency for audit + key store; dev-mode env-var shim is the only escape hatch

Audit log + API key store both require Postgres; `build_audit_sink(settings)` and the key-lookup path raise on missing connection unless dev mode is active. Two alternatives were considered. **Optional like Redis** (no-op `NullAuditSink` + file/env-driven key store as fallback) was rejected because the key store cannot be optional — it is on the auth hot path, and a "default = single env-var key" fallback recreates exactly the Phase 1 stub Phase 7 is supposed to retire. **Hard-required with no escape hatch** was rejected because it would break `uv run uvicorn ...` for new contributors who haven't stood up Postgres yet — the existing `KESTREL_DEV_API_KEY=""` empty-bypass posture made local dev frictionless and we keep that property. The chosen path: Postgres is required in any production-shaped run, AND `KESTREL_DEV_API_KEY=<non-empty>` activates a dev-mode shim that accepts that token without a DB roundtrip AND logs a loud `dev_api_key_in_use` warning on every request. Setting both Postgres and `DEV_API_KEY` is allowed and works as union (dev key accepted in addition to DB keys) — eases migration. Audit writes during dev mode go to `NullAuditSink` if no Postgres URL configured.

### 7-key-storage — Postgres is the single source of truth for API keys

API keys live in one `api_keys(id, key_hash, label, created_at, revoked_at, scopes)` table. Authentication, listing, revocation, and audit-event joins all read from the same row. Two alternatives were considered. **File + Postgres mirror** (primary store in a YAML/JSON file, revocations also written to Postgres for audit) was rejected because it makes the CLI awkward (two writes, partial-failure recovery), makes revocation racy (file edit + DB write are not atomic), and creates a second source of truth that drifts under load. **Postgres for new keys, env var as legacy single-key fallback** was rejected because that's just the dev-mode shim with a different name — codifying it as production policy invites the legacy-key path to ossify. Single source of truth keeps the model simple: one place to look, one place to write, one place to audit.

### 7-key-hash — SHA-256 unsalted over high-entropy random tokens; not bcrypt/argon2

Stored `key_hash` is `hashlib.sha256(token_bytes).hexdigest()`. Token generation is `secrets.token_urlsafe(32)` (32 random bytes → 256-bit entropy → ~43 URL-safe chars). Two alternatives were considered. **bcrypt or argon2id** (slow-by-design password hashes) was rejected because their cost is calibrated to defeat brute-force over low-entropy human passwords; an attacker who steals the hash table cannot brute-force a 256-bit random token under any time budget, so the work-factor buys nothing — and it adds latency to every authenticated request, which on Kestrel's auth hot path is unacceptable. **Salted SHA-256** was rejected because salts defeat rainbow tables — irrelevant for tokens with 256 bits of entropy, no rainbow table covers that space. The chosen unsalted SHA-256 is fast enough to be a no-op on the request path, makes the lookup query a trivial `WHERE key_hash = $1` (no per-row hash recomputation), and gives the right protection: if the `api_keys` table leaks, the hashes still don't reveal the tokens. Token generation MUST use `secrets.token_urlsafe(32)` or stronger — this is a lock; weaker entropy invalidates the SHA-256 choice.

### 7-ratelimit-storage — Token-bucket storage follows `KESTREL_SESSION_BACKEND`

Rate-limit buckets share the session backend: `KESTREL_SESSION_BACKEND=memory` → in-process per-worker buckets; `KESTREL_SESSION_BACKEND=redis` → Redis-backed buckets (same connection pool as the Redis session registry). Two alternatives were considered. **Always Redis** was rejected because it forces the Redis dep on single-worker deployments that don't otherwise need it — at odds with the Phase 4 substep 7 posture that Redis is opt-in. **Always in-memory** was rejected because multi-worker deployments would have per-worker buckets, which means a 100 req/min limit becomes effectively `100 × N_workers` req/min — the limit silently scales with worker count, which is the opposite of what an operator setting a limit expects. Following the session backend gives one configuration knob, consistent multi-worker semantics, and no new dep surface. New setting `KESTREL_RATE_LIMIT_BACKEND` is NOT introduced — it tracks `session_backend` by construction in `build_rate_limiter(settings)`.

### 7-audit-sync — Fire-and-forget bounded async queue; drop + count on overflow

Audit events are appended to an in-process `asyncio.Queue(maxsize=1000)`; a single background task drains the queue and writes batches to Postgres. The request handler does NOT await the write — it returns the response as soon as the queue accepts the event. Two alternatives were considered. **Sync write before responding** (durable: no event is lost if the process dies between handler return and DB write) was rejected because it adds DB latency to every authenticated request — for a code-execution service where a `/execute` already costs hundreds of ms of subprocess/docker work, the extra ~5ms is acceptable in isolation, but the failure mode is worse: if Postgres is slow or unreachable, every request slows or 500s — making the audit log a liveness dependency of the work endpoints. **Unbounded queue** was rejected because under sustained Postgres outage the queue grows without bound and OOMs the worker. The chosen bounded fire-and-forget keeps the work endpoints fast and resilient: queue overflow during a Postgres incident drops events, increments a `kestrel_audit_dropped_total` Prometheus counter (so operators see it on the dashboard), and the service stays up. Audit is observability, not the system of record — the contract is "best-effort durable, never blocks the request." On graceful shutdown the lifespan handler awaits a final flush of the queue with a short timeout.

### 7-admin-auth — Admin endpoints gated by `admin` scope on the API key

`api_keys.scopes` is a `list[str]` column (Postgres `text[]`). Default scope is `execute`; admin keys are created with `kestrel-keys create --scope admin <label>` and carry `["execute", "admin"]`. Admin routes (`/admin/*`) require `admin` in the requesting key's scopes. Two alternatives were considered. **Separate admin-key family** (a different table, a different auth dependency, a different env var for the admin token) was rejected because it doubles the auth surface for no benefit — the same `secrets.compare_digest` + lookup logic, written twice. **Per-route ACL** (`api_keys.allowed_routes`) was rejected as over-engineering — Phase 7 has two scopes (`execute`, `admin`); a full ACL model is premature until we have more than two. The chosen scope-flag pattern is the smallest extension to the existing auth dep: `require_api_key(scope="admin")` parameterized by the route. Future scopes (`metrics-read`, `audit-read`) slot in without schema changes.

### 7-metrics-auth — `GET /metrics` is open, like `/health`

The Prometheus scrape endpoint requires no auth. Two alternatives were considered. **Admin-scope-gated** was rejected because Prometheus scrapers typically have no credentials configured — gating `/metrics` forces operators into one of (a) embedding a bearer in the scrape config (operationally awkward, key rotation is harder), (b) putting Kestrel behind a reverse proxy that strips/adds auth, or (c) running a sidecar that does the same. **Open but firewalled by deployment** was rejected as a non-answer — every endpoint can be firewalled; the question is the default. The chosen open default matches `/health` (and Phase 1's "public diagnostic endpoints, gated work endpoints" rule from `DECISIONS.md`). Operators who want it gated can put it behind their proxy. Metric values are non-sensitive: counts, durations, gauges — no payloads, no request_ids, no api_key labels (which we explicitly avoid because high-cardinality labels blow up Prometheus storage; api_key_id is recorded in the audit log instead).

### 7-reqid-stream — `request_id` only in terminal `result` and `error` stream frames

The WebSocket streaming protocol's `StreamMessage` discriminated union gains a `request_id: str` field on the `StreamResult` and `StreamError` variants only. `StreamStdoutChunk`, `StreamStderrChunk`, and `StreamHeartbeat` do NOT carry it. Three alternatives were considered. **Every frame** was rejected because chunk frames flow at high cadence (every `print` flush) — adding 36 bytes of UUID to each chunk inflates bandwidth and stream payload size for no client benefit (the client already correlates by connection, not by frame). **Only the initial frame** was rejected because there is no initial frame — the streaming kernel emits chunks until completion; the first message a client sees could be a chunk, a heartbeat, an error, or a result. **Out-of-band (HTTP `X-Request-ID` header on the WebSocket upgrade response)** was rejected because most browser WebSocket APIs don't expose upgrade-response headers to JS — the `request_id` would be invisible to the client. Terminal frames are the natural correlation point: a client logging `result` or `error` for a completed execute has the `request_id` to grep against server logs and audit-log rows. The polling fallback's `GET /sessions/{id}/executions/{execution_id}` response gains the same `request_id` field at the top level (not per-message).

**Substep 1 implementation note (2026-05-23).** The "kernel JSON-line `request_id` field" phrasing in 7-scope is implemented **host-side only** for substep 1: the host reads the structlog `request_id` contextvar bound by the X-Request-ID middleware and attaches it to outbound `StreamResult` / `StreamError` / `PollingReadResponse` frames. The kernel does NOT receive or echo `request_id`. Reason: nothing on the kernel side currently consumes it — the host knows the id at terminal-frame construction time, and the host writes the audit row (substep 2) so the kernel doesn't need it. Avoids a kernel protocol bump and image rebuild (`:0.5.0` stays current through Phase 7 substep 1). The kernel-side extension stays available as a follow-on if substep 2's audit work surfaces a real need.

### 7-db-stack — SQLAlchemy 2.x + Alembic + asyncpg (per Brief §7.1)

Postgres access goes through SQLAlchemy 2.x's async engine layered on asyncpg, with Alembic for migrations. Three alternatives were considered. **yoyo-migrations + raw asyncpg** was the leaner option (one fewer dep, one fewer abstraction layer, hand-written SQL migrations) and was the original recommendation for substep 2 slice 1 — rejected on a spec check: the project's dependency spec explicitly names `sqlalchemy`, `alembic`, and `asyncpg` as core deps, and project convention is not to rewrite `pyproject.toml` deps without checking that spec. **alembic without SQLAlchemy** (alembic supports hand-written migrations against raw asyncpg) was rejected because losing SA gives up alembic's main draw — autogeneration from declarative models — while still paying alembic's config-file overhead. **DIY migrations** (numbered SQL files + a 30-line runner) was rejected as the kind of premature abstraction the project avoids; with 2-3 migrations expected across Phase 7 + 8 it would reinvent yoyo poorly. The chosen stack matches the spec's intent, makes autogen available for substep 3's `api_keys` table without further config, and shares one connection pool (the SA async engine wrapping asyncpg) across every DB-touching component.

### 7-db-loc — DB layer lives in `src/kestrel/db/` per Brief §5

SQLAlchemy models, the async engine/sessionmaker factory, and Alembic migrations all live under `src/kestrel/db/`. Concretely: `db/models.py` (declarative `Base` + table classes), `db/session.py` (`build_engine` + `build_sessionmaker`), `db/migrations/` (alembic env.py + script.py.mako template + `versions/` migration files). The Alembic config file `alembic.ini` sits at the repo root (alembic's convention) but its `script_location` points at `src/kestrel/db/migrations`. Two alternatives were considered. **Flat `kestrel.db_models` + `kestrel.db_session` + repo-root `migrations/`** would have kept the project's existing flat-module preference (e.g. `observability.py` at the top level rather than `observability/`) but mixes app code with operational artifacts (migrations are ops/deploy concerns) at the same depth. **Co-locate audit-specific bits under `kestrel.audit`** (e.g. `kestrel/audit/models.py`, `kestrel/audit/sink.py`) would tie the table to the feature, but substep 3's `api_keys` table doesn't belong to audit — putting them in one `db/` folder reflects that the table catalogue is a shared substrate, not an audit-specific concern. The chosen `db/` layout matches Brief §5's "Project Structure" sketch exactly, gives substep 3 a place to add `ApiKey` and substep 5's rate-limit persistence (if it ever leaves Redis) a place too. Brief §5 also sketches `db/session.py` and `db/migrations/`; this is the first time the project adopts a Brief-prescribed subdirectory beyond `api/` and `execution/`.

### 7-audit-schema-split — Two `AuditEvent` classes by design: pydantic for queue, SA `AuditEventRow` for DB

The audit pipeline carries a pydantic `AuditEvent` (in `kestrel.audit`) on the in-process queue; the Postgres sink translates it to a SQLAlchemy `AuditEventRow` (in `kestrel.db.models`) at insert time. Two alternatives were considered. **One class — SQLAlchemy only** (routes build the SA row, hand it to the sink) was rejected because it forces non-Postgres sinks to import SQLAlchemy: `NullAuditSink` would pull SA into the dev-mode hot path for no reason, and a future file/HTTP sink would import a DB ORM to write a JSON line. **One class — pydantic only** (the queue payload IS the persisted shape, with SA reading it via a JSON column) was rejected because it kills the indexed columns that make audit queries usable in production (`WHERE request_id = $1`, `WHERE api_key_id = $1`, `WHERE ts > $1`) — a single JSON column either loses these or requires Postgres `jsonb` GIN indexes which are heavyweight and operator-hostile to query. The chosen two-class split keeps the queue payload framework-free, preserves proper relational columns + indexes in Postgres, and isolates the SA dep to `PostgresAuditSink` + `kestrel.db`. The cost is the translation step in `PostgresAuditSink._insert(event)` — trivial, mechanical, ~10 lines.

### 7-postgres-test-container — `kestrel-postgres` / `postgres:16-alpine` on port 5432, skip when unreachable

Tests that exercise the Postgres path use a locally-run `kestrel-postgres` container at `postgres:16-alpine` listening on 5432, mirroring the `kestrel-redis` pattern from Phase 4 substep 7. The test fixture (added in slice 2) skips when the container isn't reachable — same posture as `redis_reachable()` in `tests/conftest.py`. Two alternatives were considered. **docker-compose for both Redis and Postgres** (Brief §8.1 sketches `docker compose up -d redis postgres`) was rejected for slice 1 because the project's existing Redis story uses manual `docker run`, and adopting compose for one service while the other stays manual creates a split convention; a compose adoption should be its own decision covering both services. **pytest-postgresql plugin** (auto-starts a Postgres process per test) was rejected because it requires Postgres binaries installed on the host, fights with WSL2's package story, and gives slower tests than a long-lived container. The chosen pattern: `docker run -d --name kestrel-postgres -p 5432:5432 -e POSTGRES_PASSWORD=kestrel postgres:16-alpine`, fixtures use `kestrel-postgres:kestrel@localhost:5432/postgres` (the default `postgres` db with the bootstrap user), tests run `alembic upgrade head` then `alembic downgrade base` per session via a fixture so schema is fresh. Adds Postgres to the "prereqs at session start" checklist alongside Docker and Redis.

**Substep-2 slice-2 implementation refinement (2026-05-25).** The originally-locked "alembic upgrade head then downgrade base per session" turned out to be needlessly expensive — downgrade runs on session teardown, which doesn't matter between test runs (next session starts with whatever schema was left). The actual implementation in `conftest.py` runs `alembic upgrade head` ONCE per pytest session (cached via session-scoped fixture), and per-test isolation is achieved by `TRUNCATE audit_events` at the start of each test inside the `postgres_engine` fixture. Faster, simpler, same isolation guarantee for tests that only look at audit_events. When substep 3 adds `api_keys`, that fixture will truncate both tables (or use `TRUNCATE ... CASCADE`). No downgrade in the test flow.

### 7.2-drain-pattern — Single-event drain loop, not batched

The `PostgresAuditSink` drain task pulls one event off the queue and inserts it in its own SQLAlchemy session: no time-window or size-window batching. Two alternatives were considered. **Time + size batching** ("flush every 50 events or every 1s, whichever first") was the original direction considered for slice 2; rejected because the complexity (timer task + size threshold + retry semantics on partial-batch failure + extra shutdown corner cases) buys nothing at Kestrel's expected request rate. A code-execution service is bounded by docker container spinup time (~hundreds of ms); even at sustained 100 req/s the audit drain only needs to handle 100 inserts/s, which asyncpg with prepared statements does comfortably without batching. **No drain task — synchronous insert in `emit()`** was rejected by the parent `7-audit-sync` lock ("never block the request"). The chosen single-event pattern keeps the drain loop trivial — `while True: event = await queue.get(); await _insert_one(event); queue.task_done()` — and leaves batching as a perf optimisation available if `kestrel_audit_dropped_total` ever shows the drain falling behind. The metric is the trigger; until it lights up, batching is premature.

### 7.2-drain-error — Drop on any drain exception; never retry inside the drain loop

When the drain task fails to insert an event (any exception during `_insert_one`), the event is dropped, `AUDIT_DROPPED` is incremented, the exception is logged via `logger.exception`, and the drain loop continues with the next event. No retry, no requeue, no on-disk overflow. Two alternatives were considered. **Retry with backoff** (catch `OperationalError`, sleep N seconds, retry the same event) was rejected because retrying inside a bounded-queue drain task is the worst-of-both: the queue keeps filling while the drain is stuck on one event, then overflow starts dropping newer events with no relation to the underlying problem — the operator dashboard becomes incoherent ("why is the drop counter going up when Postgres is back?"). **Requeue to head of queue + retry next loop** was rejected for the same reason plus head-of-queue blocking (one poison event blocks all subsequent drains). The chosen drop-on-error makes the failure mode legible: any audit problem — overflow, transient DB error, programmer bug — increments the same counter, and the underlying cause is in the logs. Audit is observability, not the system of record; the contract from `7-audit-sync` already accepts data loss under stress, so error-loss is the same shape.

**Implementation note (2026-05-25).** Slice-2 test `test_drain_survives_transient_error` caught a real gap in the first draft of `_drain_loop`: the loop awaited `_insert_one(event)` without its own try/except, on the assumption that `_insert_one` handled its own errors internally. The assumption was correct for the real `_insert_one` (it has a broad try/except around `session.commit()`), but any escapee — programmer bug, unexpected exception class, test-supplied monkey-patch that doesn't follow the convention — would kill the drain task and leave audit silently broken until process restart. The fix was a belt-and-suspenders try/except in `_drain_loop` itself that catches `Exception`, increments `AUDIT_DROPPED`, logs, and continues. Two layers of error handling for the same thing is normally a code smell; here it is the difference between "audit degrades visibly" and "audit silently stops working." The drain task is a long-running background coroutine — defending its loop body is the same hygiene that the sweeper tasks in `SessionRegistry` already practice.

### 7.2-engine-owner — Lifespan owns the SQLAlchemy engine; sink takes it as a constructor parameter

The FastAPI lifespan in `create_app()` builds the async engine (`build_engine(settings)`) when `settings.audit_backend == "postgres"` and passes it into `build_audit_sink(settings, engine=engine)`. The sink stores the engine reference and builds a sessionmaker from it, but does NOT call `engine.dispose()` — the lifespan does that, in the `finally` block after `audit_sink.aclose()`. Three alternatives were considered. **Sink owns engine** (`PostgresAuditSink.__init__` calls `build_engine` itself; `aclose` calls `engine.dispose`) was rejected because substep 3's API-key store will need the same engine — if audit owns it, substep 3 either duplicates the pool (operationally bad: two pools, double the connections, no shared cached statements) or pulls the engine out of audit (weird coupling — keys depending on audit). **Module-level engine singleton** (`get_engine()` in `kestrel.db.session` with `@lru_cache`) was rejected because process-wide singletons make test isolation harder (which engine does the test see? the cached one or a fresh one?) and they have to dance with the FastAPI lifespan for clean shutdown. The chosen lifespan-owned engine + injected sink keeps the dependency graph explicit and lets multiple components (substep 2 audit, substep 3 keys, anything later) share one pool with zero plumbing. Factory signature is `build_audit_sink(settings: Settings, engine: AsyncEngine | None = None) -> AuditSink`; passing `engine=None` when `audit_backend == "postgres"` raises `ValueError` (fail-fast on caller error rather than silent fallback to `NullAuditSink`, which would silently lose audit data).

**Lifespan unwind logic (2026-05-25).** The slice-1 `app.py` lifespan started the audit sink *after* the session registry, but didn't account for what happens if audit start fails after registry start succeeded. The slice-1 design review flagged this; slice 2 fixes it: the `build_audit_sink + audit_sink.start()` pair is wrapped in `try/except` that calls `await registry.aclose()` (and `await engine.dispose()` if the engine was built) before re-raising. The success-path teardown order in `finally` stays audit → engine → registry: audit closes first to drain its queue using the engine, then engine releases the asyncpg pool, then registry releases its kernels. Order matters because audit uses the engine; reversing would dispose the engine while the drain task is still trying to use it.

### 7.3-emit-locus — Per-route emit, not middleware; explicit audit gaps accepted for slice 3

Audit events are emitted by route handlers (for HTTP routes) and by the polling orchestrator (for polling executions). The HTTP middleware does NOT emit. Three alternatives were considered. **Middleware-only emit** (the request-ID middleware also writes audit) was rejected because the middleware doesn't naturally know route-specific fields (`code_length`, `exit_code`, `timed_out`, `session_id`, `execution_id`); routes would have to stash data onto `request.state.audit_extras` for the middleware to pick up — workable but introduces a magic side channel and makes per-route audit logic invisible at the call site. **Middleware emit for catch-all + route emit for enrichment** (both layers emit, second row replaces first) doubles the row count and complicates the "one row per request" mental model. **Decorator-based emit** (each route decorated with `@audit_route("/x")`) was rejected because it would have to inspect args at decorator-eval time, fragile across the heterogeneous route shapes (HTTP success/exception, WS post-accept, polling orchestrator). The chosen per-route emit is verbose but explicit — every audit row's construction is visible in the handler that produced it, easy to reason about, easy to test. The cost is documented gaps: auth failures (require_api_key raises 401 before any handler runs), validation errors (FastAPI's 422 path), pre-accept WebSocket rejections (4401, pre-accept 4404), and polling POST 404s (session not found before orchestrator spawn) all produce no audit row. For those, operators rely on the `kestrel_http_requests_total{status="401"|"404"|"422"}` Prometheus counter. Substep 4 or later can revisit if security-observability requirements demand audit rows on those paths (e.g. by adding emit to `require_api_key` itself, or by introducing the middleware-with-extras pattern).

### 7.3-status-mapping — Centralized `http_status_for_exception` helper in `kestrel.audit`

The exception → HTTP status mapping used by audit emit is `kestrel.audit.http_status_for_exception(exc)` — one function, one source of truth. Two alternatives were considered. **Inline per route** (each route's exception-branch hardcodes `404` for `SessionNotFound`, `409` for `SessionBusy`, etc.) was rejected because the same five-class switch would appear in 5+ places and drift when new exceptions are added. **Map on the exception class itself** (e.g. `SessionNotFound.HTTP_STATUS = 404` as a class attribute) was rejected because it requires modifying the execution modules to encode HTTP concerns, which they otherwise know nothing about — wrong direction of coupling. The chosen helper centralizes the switch in `kestrel.audit`, which becomes a small bridge module: it imports from `kestrel.execution.session_registry` and `kestrel.execution.session_runtime` for the type checks (done inside the function via lazy import to keep `kestrel.audit` lightweight at module load). The slice-1 `kestrel.audit`'s "leaf module" property is relaxed by one degree — it now knows about session-related exception classes, but only at function-call time, not at import time. Future Phase 7 substeps that add new exception classes (e.g. substep 3's `ApiKeyRevoked`) extend the helper in one place.

### 7-key-loc — API-key store lives at `src/kestrel/api_keys.py` (top-level module)

The Postgres-backed API-key store + `ApiKeyStore` Protocol + helpers all live in `src/kestrel/api_keys.py` — a top-level module, mirroring the `kestrel.audit` layout. Three alternatives were considered. **Folded into `src/kestrel/api/auth.py`** would have co-located the store with `require_api_key`, but that file is the FastAPI dependency (one small function); mixing it with the full store + helpers + Protocol makes the auth dep harder to scan, and substep 4's CLI would import from a deep `kestrel.api.auth` path that doesn't reflect that the store is a domain-level concept. **New top-level module `src/kestrel/auth.py`** would have collided naming-wise with the existing `kestrel.api.auth` family (auth-dep vs auth-store, both called "auth" — confusing). **A new subpackage `src/kestrel/api_keys/`** with `store.py`, `helpers.py` etc. was rejected as over-engineering — slice 1 + 2 fit comfortably in one file (~180 lines including docstring), and the slice-2 wiring layer for `require_api_key` belongs in `kestrel.api.auth` anyway. The chosen single-file top-level mirrors `kestrel.audit` exactly: domain module that the FastAPI layer (`kestrel.api.auth` for keys, `kestrel.api.routes`/`sessions.py` for audit) consumes via Protocol injection. Substep 4's `kestrel-keys` CLI imports `from kestrel.api_keys import ...` cleanly.

### 7-key-token-prefix — Tokens carry a `kestrel_` identifier prefix

Generated tokens look like `kestrel_<43-char-url-safe-base64>` — `secrets.token_urlsafe(32)` per `7-key-hash` plus a 7-character `kestrel_` prefix. Two alternatives were considered. **Raw `secrets.token_urlsafe(32)`** (no prefix) was the literal reading of `7-key-hash`, rejected because it makes tokens indistinguishable from any other random base64 blob; a leaked token in a paste or log can't be auto-identified as a Kestrel credential. **Service-version-encoded prefix** like `kestrel_v1_<...>` was rejected as premature — there's no v2 to disambiguate from, and we can extend later if needed. The chosen plain `kestrel_` prefix matches industry practice (GitHub's `ghp_`/`gho_`, Stripe's `sk_test_`/`sk_live_`, AWS's `AKIA...`): the prefix is publicly known, contributes zero security, but lets secret-scanning tools (GitHub secret scanning, GitGuardian, internal grep-based hunts) identify leaked tokens by pattern without entropy analysis. Cost: 7 bytes per token. Benefit: every leaked token is auto-classifiable + revocable. The hash stored in `api_keys.key_hash` is computed over the full prefixed string, so the prefix is part of the secret — partial-prefix attacks (e.g. "I leaked just the prefix") are not a thing because `kestrel_` alone won't verify against any row.

### 7-key-null-store-as-none — `api_key_backend="null"` returns `None`, not a `NullApiKeyStore`

When `settings.api_key_backend == "null"`, `build_api_key_store(settings)` returns `None`. There is no `NullApiKeyStore` class. Two alternatives were considered. **Audit-pattern symmetry**: `NullApiKeyStore` with no-op `verify`/`create`/`revoke`/`list` methods, mirroring `NullAuditSink`'s shape. Rejected because most store methods can't be no-ops sensibly — `create()` returning a fake token is dangerous (the caller thinks they got a real key), `revoke()` returning True for a fake id is misleading, `list()` returning empty is at least honest but `verify()` returning None on every input means auth always fails which is a different policy from "auth disabled". **`MemoryApiKeyStore`** (an in-memory implementation for dev mode) was rejected as out of scope for slice 1 — substep 4 may add it if real demand surfaces, but Phase 7 ships fine without it. The chosen `Optional[ApiKeyStore]` pattern keeps the auth flow honest: `require_api_key` (slice 2) explicitly checks `if store is None` and branches to the legacy dev-key path. The branch is one line; the alternative would be a 40-line stub class. Callers also benefit: a future ApiKey CLI (substep 4) that finds `store is None` can refuse to operate ("Postgres backend required") rather than silently doing nothing against a fake store.

### 7-auth-flow-order — Dev shim first, then store, then 401; auth disabled only when both empty

`require_api_key` (and the WS auth handler) resolves a bearer in this exact order:

1. If `settings.dev_api_key == ""` AND `store is None` → return `None` (auth disabled). Phase 1-6 backward-compat path; preserves "empty dev key = no auth required" for tests that never opted into the store.
2. If the bearer is missing or empty → 401.
3. If `settings.dev_api_key` is set AND the bearer matches it (via `secrets.compare_digest`) → log `dev_api_key_in_use`, return `"dev"`. This is the per-`7-pg-required` operator escape hatch.
4. If `store is not None` → call `store.verify(bearer)`. If a row matches (not revoked) → return the `ApiKeyInfo`.
5. Otherwise → 401.

Three alternatives were considered. **Store first, dev shim fallback** would hash + DB-lookup *every* request before checking the env-configured shim, costing one wasted DB roundtrip on every dev-mode request. **Dev shim only when store unavailable** (don't even check `dev_api_key` if the store exists) was rejected because it forces operators to choose between "Postgres mode" and "I have a local dev key for debugging" — the whole point of `7-pg-required` is they coexist. **No auth-disabled fallback** (always 401 when nothing matches) was rejected because it breaks the Phase 1-6 backward-compat property: an existing deployment that ran with empty `dev_api_key` to disable auth would suddenly start rejecting everything. The chosen order makes the dev path fast (one constant-time compare, no DB), the store path correct (full hash + lookup), and the disabled path explicit (both knobs must be empty). Inside the implementation, this is split across the `verify_bearer` helper (returns the resolved identity or raises `AuthRejected`) and the transport-specific wrappers (`require_api_key` → HTTP 401, WS handler → close 4401).

### 7-dev-shim-sentinel — Dev-shim path returns the literal string `"dev"`, not a synthesized `ApiKeyInfo`

When the dev shim accepts a request, `verify_bearer` returns Python's literal string `"dev"`. Audit emit sites then store `api_key_id = "dev"` in the audit row. Two alternatives were considered. **Synthesized `ApiKeyInfo`** with a fixed sentinel UUID (e.g. `uuid.UUID("00000000-0000-0000-0000-000000000001")`) and a literal label `"dev-shim"` was rejected because it makes the dev key look like a "real" key in the audit log — operators querying `WHERE api_key_id = '<sentinel-uuid>'` would have to memorize the sentinel; a future schema migration that joins `audit_events.api_key_id` to `api_keys.id` would find no row (the sentinel doesn't exist in the table), producing confusing nulls. **A separate `audit_events.dev_shim_used` boolean column** was rejected as schema bloat for a niche path. The chosen string-sentinel is honest: `api_key_id = 'dev'` is unambiguously NOT a UUID, the storage type (`String(64)`) accepts both, the dashboard / SQL query `WHERE api_key_id = 'dev'` is the natural filter for "dev-shim activity", and there's no fake row to maintain. The return-type-union on `verify_bearer` (`ApiKeyInfo | str | None`) is slightly less elegant than a single dataclass, but the `audit_id_for(info)` helper in `kestrel.api_keys` papers over the type check at every call site.

### 7-auth-shared-helper — `verify_bearer` + `AuthRejected` as a transport-neutral resolver

Authentication logic lives in `kestrel.api.auth.verify_bearer(bearer, settings, store)` — a single async function that returns the resolved identity or raises `AuthRejected`. Both the HTTP path (`require_api_key`) and the WebSocket path (`sessions_stream.execute_in_session_stream`) call it and translate the rejection to their transport's idiom (HTTP 401 vs WS close 4401). Two alternatives were considered. **Duplicated logic** — `require_api_key` does its own check, the WS handler does its own check (the pre-slice-2 shape with `_auth_ok`) — was the status quo and fine while auth was a one-line `secrets.compare_digest` against a single env var; substep 3's dev-shim-then-store flow has too many branches to safely duplicate. **A middleware-style auth layer** that runs before any handler and stashes the result on `request.state` was rejected because it doesn't fit WS (no HTTP middleware) and because the cache-hit semantics of `Depends(require_api_key)` already handle "run once per request" for HTTP. The chosen helper is plain async + raises. `AuthRejected` is a fresh exception class in `kestrel.api.auth` — narrower than reusing `HTTPException(401)` (which is HTTP-specific) and narrower than `Exception` (which catches too much). The HTTP wrapper raises `HTTPException(401, ...)`; the WS handler closes with code 4401; both go through the same `verify_bearer` body.

### 7.3-ws-method-label — WebSocket audit rows use `method="WS"`

For WebSocket-route audit rows, `method` is the literal string `"WS"`. Two alternatives were considered. **`"GET"`** (the actual HTTP method of the upgrade handshake) loses the WebSocket-ness — an operator querying for "all WS streaming activity" would see only `WHERE route LIKE '%execute/stream%'`, with `method` always equal to the unhelpful `"GET"`. **`"WEBSOCKET"`** (more descriptive) was rejected for terseness and to fit the existing `String(16)` column size with margin. The chosen `"WS"` is a custom non-standard marker that makes WebSocket rows trivially identifiable. The `method` column accepts arbitrary strings (the `String(16)` schema doesn't enforce HTTP verb semantics), so this is purely a Kestrel convention. Substep 3+ admin endpoints can pivot to `method = 'WS'` as their primary "show me streaming sessions" filter.

### 7-rate-limit-dims — Buckets per `(api_key_id, route_class)` with three route_classes

Three `route_class` values: `execute` (covers `POST /execute` + `POST /sessions/{id}/execute` + `POST /sessions/{id}/execute/polling` + `WS /sessions/{id}/execute/stream`), `session_lifecycle` (covers `POST /sessions`, `GET /sessions`, `GET /sessions/{id}`, `DELETE /sessions/{id}`), and `admin` (covers `/admin/*`). Two alternatives were considered. **Per `(key, route)`** (one bucket per individual route) was rejected as cardinality explosion with no model benefit — `GET /sessions` and `GET /sessions/{id}` have effectively the same cost profile; treating them separately gives the operator one knob per route instead of one knob per cost-class. **Per `key` only** (one bucket regardless of route) was rejected because cheap session-lifecycle reads (GET /sessions, listings) would compete with expensive `/execute` calls — a UI doing a sessions-list-refresh every second would eat a user's execute budget. Three classes is the minimum-useful granularity: separates the load-bearing expensive path (`execute`) from cheap reads (`session_lifecycle`) from admin operations (`admin`). New settings: `KESTREL_RATE_LIMIT_EXECUTE_PER_MINUTE` (default 60), `KESTREL_RATE_LIMIT_SESSION_LIFECYCLE_PER_MINUTE` (default 300), `KESTREL_RATE_LIMIT_ADMIN_PER_MINUTE` (default 60). Token-bucket capacity = per-minute limit, refill rate = limit / 60 per second. 429 response includes `Retry-After: <seconds-until-next-token>`.

### 7.4-url-source — `kestrel-keys` reads the database URL from `KESTREL_DATABASE_URL` only, no `--database-url` flag

The CLI builds its `Settings()` from the process environment, identical to how the running service does. There is no `--database-url` command-line override. Two alternatives were considered. **`--database-url` flag** (with env fallback) was rejected because URLs carry passwords; flag-style invocation puts the secret on the operator's shell history and process list. **Implicit `localhost` default** (no env required) was rejected because Phase 7 is explicitly multi-host — the dev box's Postgres and the operator's Postgres are different machines, and a silent default is the most footgun-prone option. The chosen env-only path makes the CLI a thin client over the same configuration surface as the server: if the operator can `KESTREL_DATABASE_URL=... uv run uvicorn ...`, they can `KESTREL_DATABASE_URL=... uv run kestrel-keys ...`. When the env var is empty, the CLI prints a clear error and exits 1 — same shape as the audit-postgres factory's `ValueError`.

### 7.4-list-format — Default human table + `--json` opt-in for scripts

`kestrel-keys list` prints a fixed-width text table by default; passing `--json` flips to a JSON array on stdout. Two alternatives were considered. **JSON by default** was rejected because the primary user is a human operator running the command interactively; piping into `jq` for everyday inspection is friction. **Table only** (no JSON) was rejected because the second-most-common use is shell automation ("revoke every key older than 90 days"), where parsing fixed-width columns is brittle. The two-mode shape is standard CLI practice (e.g. `kubectl get … -o json`). The JSON contract is the stable interface: a JSON array of `{id, label, scopes, created_at, revoked_at}` objects where `revoked_at` is either an ISO-8601 string or `null`. The table format is explicitly NOT a contract — tests assert only on the JSON path; humans tolerate column-width drift.

### 7.4-revoke-target — `revoke <id>` takes a UUID, not a label

The `revoke` subcommand accepts exactly one positional argument: the key's UUID. Two alternatives were considered. **Revoke by label** (`kestrel-keys revoke --label ci-test-key`) was rejected because labels are not unique in the `api_keys` schema by design — operators frequently rotate keys with the same label (`ci-test-key` v1, v2, v3 all coexist during cutover), and revoking by label would be ambiguous. **Revoke by token-prefix** (the first 12 chars of the plaintext token) was rejected because the plaintext is never recoverable from the DB after `create`, so this would require operators to keep a separate ledger of token prefixes — exactly what the DB is supposed to be. The chosen UUID-only path forces a `kestrel-keys list` lookup first, which is the right ergonomic: the operator sees the full row (label, scopes, created_at) before destroying access. The argparse error on a malformed UUID returns exit code 2; revoking an unknown or already-revoked id also returns exit 2 with an explanatory stderr message.

### 7.4-scope-shape — `--scope` is a repeatable flag, not a comma-separated value

`kestrel-keys create my-key --scope execute --scope admin` is the supported invocation. There is no `--scopes execute,admin` form. Two alternatives were considered. **Comma-separated** (`--scopes`) is more compact but introduces quoting subtleties — `--scopes execute, admin` (stray space) silently produces a scope literally named `" admin"`. **Single `--scope` with internal split** has the same problem. The chosen `action="append"` form is the standard argparse idiom (mirrors `--volume` in `docker run`, `--header` in `curl`), gets shell quoting for free, and is unambiguous about empty values. When `--scope` is omitted entirely, the default `["execute"]` from `PostgresApiKeyStore.create` applies — the CLI passes `None` through rather than re-encoding the default, so the source of truth for the default scope stays in one place.

### 7.4-test-style — Tests invoke the CLI as a real subprocess via `uv run kestrel-keys`

Test cases shell out: `subprocess.run(["uv", "run", "kestrel-keys", ...], capture_output=True, text=True)`. Two alternatives were considered. **`from kestrel.cli.keys import main; main(["create", "label"])`** is faster (~10ms vs ~1s per test) and gets coverage data, but it doesn't exercise the console-script wrapper installed by hatch — a broken `[project.scripts]` entry would pass the import-based test and break in production. **A `CliRunner`-style harness** (à la Click's testing utilities) is overkill for argparse and adds a new dep. The chosen subprocess approach is slow per test (three tests total → ~3s) but tests the actual operator invocation end-to-end: the wrapper script, the env-var pickup, the asyncio.run boundary, the exit code. Tests still share the per-test `postgres_engine` fixture (which TRUNCATEs `api_keys`), so the subprocess and the assertion-side store both see the same DB state — cross-process visibility is just "they both connect to localhost:5432".

### 7.5-unauth-skip — Rate limiting is skipped entirely when the request has no identity

When `require_api_key` returns `None` (auth-disabled path: `dev_api_key == ""` AND `api_key_backend == "null"`), the rate-limit dependency is a no-op — the request passes without consulting any bucket. Three alternatives were considered. **Per-client-IP bucket** for anonymous requests is closer to real-world rate limiting but introduces a second concept (X-Forwarded-For trust chain, CIDR aggregation, IPv6 prefix grouping) that isn't justified at this stage. **Single global `"anonymous"` bucket** is simple but means one dev test run can exhaust the bucket for every other dev process sharing the worker, which is the opposite of what dev mode wants. **Reject all unauthenticated requests with 429** would break Phase 1-6 backward compat. The chosen "skip" path keeps dev/test friction at zero, and the operational mitigation is the existing "don't run prod with an empty dev key" rule — the dev-shim log line (`dev_api_key_in_use`) already surfaces that, and Phase 7 substep 3's locked behavior of `7-pg-required` makes the "no key store + empty dev key" path explicitly the dev-only escape hatch. The dev shim itself IS rate-limited (see `7.5-dev-shim-limit`); only true no-identity requests skip.

### 7.5-dev-shim-limit — The dev shim is rate-limited with `"dev"` as the bucket identity

When `verify_bearer` returns the dev-shim sentinel `"dev"` (decision `7-dev-shim-sentinel`), the rate-limit dependency uses the literal string `"dev"` as the identity, sharing one bucket across all dev-shim usage. Two alternatives were considered. **Skip the limit entirely for dev shim** (treat dev like the auth-disabled case from `7.5-unauth-skip`) was rejected because the dev shim is an explicit opt-in for *production* deployments that want a short-circuit (per `7-pg-required`); leaving it unbounded means a misconfigured CI job that hammers the dev endpoint can take out the whole worker. **Per-source-IP buckets for dev** would force the X-Forwarded-For machinery from `7.5-unauth-skip`'s rejected alternative back in. The chosen single-bucket-keyed-on-"dev" matches the audit convention (`api_key_id = "dev"` rows are already a single SQL filter) and gives the operator one knob to tune the dev-shim allowance — usually the same as a regular `execute` bucket, which is the right shape because the dev shim *is* a single key from the system's perspective.

### 7.5-ws-close-code — WebSocket rate-limit-exceeded closes with code 4429

WS doesn't have a 429 status code. When the rate-limit check fails before `await websocket.accept()`, the handler closes with code 4429 (Kestrel's custom-4xxx convention) instead of 1008 (RFC 6455 "policy violation"). Two alternatives were considered. **1008 (policy violation)** is the closest RFC-standard code but is too generic — clients can't tell rate-limit denial from any other policy rejection, and the existing Kestrel close codes (4401 auth, 4404 not-found, 4409 conflict, 4410 gone) all use the 4xxx range for transport-layer specifics. **Accept-then-close** with a typed `StreamError` frame would give the client a richer payload but adds a round-trip and breaks the symmetry with HTTP's pre-handler 429. The chosen 4429 mirrors HTTP's 429 status numerically (drop the leading "4xx" prefix from the status, prepend "4" for "Kestrel-custom"), which makes the close-code → HTTP-equivalent mapping mechanical. Clients already have to handle 4401/4404/4409/4410; 4429 fits the same handler. The pre-accept close means no audit row is emitted for these (audit gap documented in `7.3-emit-locus`); the `kestrel_rate_limited_total{route_class="execute"}` counter is how operators see the volume.

### 7.5-ws-per-execute — WebSocket charges one token per execute message, not per connection

A WS stream is a long-lived connection on which the client can run many execute messages. Each execute message deducts one token from the `execute` bucket; the connection itself is free. Two alternatives were considered. **Per-connection charge** (deduct one token at `accept`) is simpler but mis-charges the workload: a chatty client with one socket running 100 executes pays the same as a thoughtful client with one socket running one execute and then idling, even though their server cost differs by 100x. **Per-byte / per-second** would track real resource cost more accurately but requires a metering layer that doesn't exist and is a poor fit for token-bucket math. The chosen per-execute-message charge matches what HTTP does (one HTTP request to `/sessions/{id}/execute` = one token), keeps the `execute` route_class meaning consistent across all four transports (POST stateless, POST session, POST polling, WS stream), and means a session-pooling client that opens one socket and runs 60 small executes hits the same limit as one that opens 60 sockets — fairness without forcing the client to choose between connection patterns. The token is deducted at the same point in the kernel-execute flow where the audit row is queued, so accounting and observation stay aligned.

### 7.5-metric — Dedicated `kestrel_rate_limited_total{route_class}` counter

A new Prometheus counter `kestrel_rate_limited_total` with one label `route_class` (cardinality = 3) is added to `kestrel.observability`. Two alternatives were considered. **Rely on `kestrel_http_requests_total{status="429"}`** (the existing per-route metric) was rejected because the existing metric uses `route` as its label, not `route_class` — operators asking "are we hitting limits often on session_lifecycle?" would have to aggregate four routes manually, and the answer would still mix actual rate-limit 429s with any other source of 429 the framework might surface in future. **A high-cardinality `{route_class, api_key_id}` counter** that names the abuser was rejected on the same label-cardinality discipline that excludes `api_key_id` from all metrics (decision `7-metrics-auth`); the audit log is the right place to identify *which* key is hitting limits. The chosen single-label counter is small, focused, and answers exactly the operator's primary question. WebSocket rejections (close 4429) bump the counter at the same point HTTP rejections (status 429) do — symmetric across transports.

### 7.5-redis-atomicity — Single Lua script via `EVAL` for the token-bucket read-modify-write

The Redis-backed rate-limit check is a read-modify-write: read `(tokens, last_refill_at_ms)`, compute refill from elapsed time, decide allow/deny, write back updated state. Three alternatives were considered. **Pipelined `HMGET` + compute + `HSET`** is two round-trips but introduces a TOCTOU race — two concurrent checks from the same identity can both read the same `tokens` value, both decide allow, and both write back `tokens - 1`, granting one more request than the bucket permits under contention. **`WATCH`/`MULTI`/`EXEC` optimistic locking** is atomic but retries on contention; under load (precisely when rate limiting matters most) every check pays the retry cost, and the failure mode is opaque to callers. **An atomic `INCRBY`-style counter** doesn't fit token-bucket — the "compute refill from elapsed time since last touch" step is not a simple counter operation; it depends on per-bucket state that `INCRBY` doesn't expose. The chosen single Lua script via `EVAL` (or `EVALSHA` after the script is cached server-side) gives full atomicity in one network round-trip — Redis runs Lua single-threaded, so the script's read/decide/write sequence is indivisible from any other Redis command's perspective. Standard recipe; Stripe's rate-limit blog, AWS API Gateway's published architecture, and GCRA reference implementations all use this pattern. The script is short (~25 lines), inlined as a Python string constant in `rate_limit.py` (no separate `.lua` file to package), and registered via redis-py's `client.register_script(...)` which auto-handles the EVALSHA caching + NOSCRIPT-fallback dance. The TTL is set inside the script (one `EXPIRE` call) so idle buckets auto-clean — no separate sweeper needed for the Redis backend.

### 7.5-redis-pool-share — `RedisRateLimiter` shares the session registry's Redis client

The Redis-backed limiter consumes the same `redis.asyncio.Redis` instance that `RedisSessionRegistry` already opens, rather than constructing a second client pointed at the same URL. Two alternatives were considered. **Separate client per component** (the limiter calls `Redis.from_url(settings.redis_url, ...)` itself) is architecturally cleaner — each component fully owns its lifecycle — but doubles the connection count to Redis (sessions pool + limiter pool, each defaulting to ~50 connections in redis-py, against the Redis default `maxclients=10000` but still wasteful at multi-worker scale). **Operator-supplied client** (a new architecture where the lifespan owns a single client and hands it to every Redis-consuming component) was rejected as scope creep — too big a refactor for slice 2 when the existing session-registry-owns-the-client pattern already works. The chosen shared-pool approach mirrors the lock spirit of `7-ratelimit-storage` (sessions and rate limits use the same backend, share the same connection story). Mechanism: `RedisSessionRegistry` exposes a new public `client` property; the lifespan reads it via `getattr(registry, "client", None)` (returns `None` for `InMemorySessionRegistry`, which has no Redis) and passes it to `build_rate_limiter(settings, redis_client=...)`. The limiter does NOT own the pool — its `aclose()` only logs; the actual `client.aclose()` happens during `registry.aclose()`. The close order in the lifespan (`rate_limiter.aclose()` BEFORE `registry.aclose()`) preserves the invariant that the limiter stops using the client before the client's owner closes it.

### 7.5-redis-time-source — Lua script takes `now_ms` from the worker, not `redis.call('TIME')`

The token-bucket refill formula needs a "now" timestamp inside the Lua script. Two alternatives were considered. **`redis.call('TIME')` inside the script** is theoretically more correct for multi-worker — a single clock source (the Redis server's clock) eliminates NTP skew between workers — but tests cannot inject a controllable clock, forcing reliance on `asyncio.sleep(N)` for refill assertions (slow tests, occasional flakiness from event-loop scheduling jitter). **Tolerating asyncio.sleep-based tests** was considered as an acceptable tradeoff for cross-worker correctness, but Kestrel's existing test suite leans heavily on deterministic clocks (the `_Clock` helper in `tests/integration/test_rate_limit.py` was added precisely for this reason), and the NTP-skew risk in practice is bounded to single-digit milliseconds — well under the 1-second granularity of the `Retry-After` HTTP header. The chosen client-supplied `now_ms` matches `InMemoryRateLimiter`'s existing `time_source` keyword — both backends accept the same parameter with the same semantics, and tests pass a `_Clock.now` callable for deterministic refill behavior. The defaults differ: `InMemoryRateLimiter`'s default is `time.monotonic` (single-process, monotonic-clock immunity to wall-clock jumps); `RedisRateLimiter`'s default is `time.time` (wall-clock Unix-epoch seconds — same reference frame across workers, so stored `last_refill_at_ms` values are comparable when different workers update the same bucket). The default difference is the only visible asymmetry between the two backends' constructors.

### 7.5-fail-policy — Rate limiter fails OPEN on `RateLimiterUnavailable`

When `RedisRateLimiter.check()` raises `RateLimiterUnavailable` (Redis unreachable, Lua script error), the per-route-class HTTP/WS dep allows the request through, logs a structured warning, and bumps a dedicated counter `kestrel_rate_limit_failures_total{route_class}`. Two alternatives were considered. **Fail-closed** (re-raise as HTTP 503 / WS close 1011) was rejected because it makes the rate limiter a hard dependency for all traffic — a Redis hiccup would take the entire service offline, including paths that have nothing to do with rate limiting like `/health`. Worse, an attacker who can DOS Redis effectively gets a denial-of-service amplifier; the limit is supposed to *protect* the service, not be the brittlest link in it. **Silent allow with no instrumentation** was rejected because the operator deserves to know when the limiter is degraded — silently letting unlimited traffic through has security implications operators need to be paged on. The chosen fail-open + dedicated counter posture treats rate limiting as a **fairness gate, not a security gate**: the system stays up under partial-failure, the operator sees the counter climb, and the alerts they configure on `rate(kestrel_rate_limit_failures_total[5m]) > 0` are the trigger to fix the storage backend. The new counter is single-label (`route_class`, cardinality 3) and lives alongside `RATE_LIMITED` in `kestrel.observability`. Slice-3 unit tests can exercise the fail-open path by monkey-patching `RateLimiter.check` to raise `RateLimiterUnavailable` — no Redis required.

### 7.5-polling-get-class — Polling `GET /executions/{id}` uses `session_lifecycle`, not `execute`

The HTTP polling fallback has two endpoints: `POST /sessions/{id}/execute/polling` (starts an execute on a background task) and `GET /sessions/{id}/executions/{execution_id}?wait=N` (long-polls for output, holding the request up to 30 seconds). The POST is unambiguously an execute and consumes from the `execute` bucket. The GET is the question. Two alternatives were considered. **`execute` class for the GET** would treat the read as part of the execute machinery, but long-polling clients typically issue 1 GET per execute (the simplest client just waits for the result, doesn't poll repeatedly), so every async-poll execute would cost 2 tokens — the POST + the GET — silently halving the operator's intended `execute_per_minute` budget. **A new fourth class** like `poll_read` (separate budget) was rejected as unjustified granularity — a polling-read is fundamentally a stateful read, the same kind of operation `GET /sessions/{id}` is, and it should share that budget. The chosen `session_lifecycle` mapping reflects the read-not-execute semantics, keeps the route-class enumeration at three (matching the lock from `7-rate-limit-dims`), and means a client doing async-poll-style execution sees one token deducted per execute, not two. The 300/min default for `session_lifecycle` accommodates the worst-case "client polls aggressively every second" pattern without surprising operators.

### 7.5-ws-check-locus — WS rate-limit check fires per-message inside the loop, not pre-accept

Decision `7.5-ws-per-execute` locks "one token per execute message, not per connection." This decision pins down where the check actually runs: **inside the WebSocket handler's per-message loop, just before `runtime.execute_stream(code)` is invoked**, NOT before `await websocket.accept()`. Two alternatives were considered. **Pre-accept check** would close 4429 before the upgrade completes — symmetric with HTTP's pre-handler 429 — but it would deduct a token at connection time, which directly contradicts `7.5-ws-per-execute`. A long-lived but mostly-idle WS connection would be charged for opening, even if it never runs an execute. **Both pre-accept AND per-message** (defense in depth) is wasteful — the pre-accept check is the wrong semantic and adds no value the per-message check doesn't already give. The chosen per-message-only path means a client can open a WS, sit idle, and pay zero; each execute message they send consumes one token; if they exhaust their bucket mid-stream, the next message triggers close 4429 (the whole socket dies, not just that message — the protocol has no "rejected this one execute, try later" frame, and adding one is scope creep). The check sits between the parsed `execute_request` and the `runtime.acquire_for_execute` call, in the same per-message critical section as audit and back-pressure. On denial: bump `kestrel_rate_limited_total{route_class="execute"}`, set audit_status=429 + audit_error_kind="rate_limited", close with WS code 4429 per `7.5-ws-close-code`, return from the handler (which fires the audit emit in the outer finally).

### 7-admin-dev-shim — Dev-shim and auth-disabled both pass the admin scope check

Substep 6's `/admin/*` routes need a scope gate, and `require_api_key` already resolves identity into three shapes: `None` (auth disabled), `"dev"` (dev shim matched), and `ApiKeyInfo` (store-verified). The rule chosen is: `info is None` → pass, `info == "dev"` → pass, `ApiKeyInfo` with `"admin"` in `scopes` → pass, else HTTP 403. Two alternatives were considered. **Strict-only-real-admin-keys** (dev shim 403s) was rejected because the dev shim already exists as a development escape hatch (decision `7-pg-required` ships the shim precisely so devs don't need to spin Postgres up for every fast iteration); forcing real keys for admin specifically would create asymmetric friction where everything else works without DB but admin endpoints don't. **Tiered-pass** (dev shim allowed but auth-disabled 403s) was rejected as incoherent — if auth itself is off because nothing is configured, gating a sub-check above it is meaningless; the operator who runs with both `dev_api_key=""` and `api_key_backend=null` is explicitly opting into the unsecured single-user mode. Production discoverability is preserved by `7-pg-required`: a prod deployment requires `KESTREL_API_KEY_BACKEND=postgres` AND `KESTREL_DEV_API_KEY=""`, and in that posture only real `ApiKeyInfo` rows with `admin` in their `scopes` array pass — the dev path simply cannot match.

### 7.6-audit-query-loc — Audit read queries live in `src/kestrel/db/queries.py`, not on `PostgresAuditSink`

`GET /admin/audit` needs to read from the `audit_events` table — a *read* path that doesn't fit `PostgresAuditSink`, which is conceptually a write-only pipeline (bounded queue + background drain + insert-one-at-a-time per `7.2-drain-pattern`). Three locations were considered. **On `PostgresAuditSink` directly** (add `list_events` method to the sink) was rejected because it conflates read and write concerns on a class whose whole shape is geared around the queue; it also forces any future non-Postgres sink (file, HTTP) to implement read too, even though those backends would have nothing to read. **A new `AuditQueryAdapter` Protocol** with `NullAuditQueryAdapter` + `PostgresAuditQueryAdapter` was rejected as over-engineered: audit has exactly two sinks (null + postgres), the null backend has no data to query, so the Protocol would have one real implementation. **Free functions in `src/kestrel/db/queries.py`** is the chosen path: a tiny new module that takes an `async_sessionmaker` and runs the query, with one function per admin read. Future admin queries (e.g. session usage stats, key activity counters) can grow this file by addition, not by widening any single class. The router consumes the sessionmaker via a new `get_sessionmaker(request)` DI provider — the lifespan already builds one when an engine is built (audit-postgres or api-key-postgres). `NullAuditSink` paths don't have a sessionmaker; the audit GET returns HTTP 503 when `audit_backend == "null"`.

### 7.6-audit-pagination — Cursor pagination via `?before_ts=<iso>`, not `?offset=N`

`GET /admin/audit` returns rows in `ts DESC` order. The list could grow into millions of rows over a long-running deployment, so the pagination shape matters for read performance, not just ergonomics. Two alternatives were considered. **Offset pagination** (`?limit=50&offset=200`) is the simplest possible thing, but every skipped row costs the query a row visit even with the index — at 1M rows and `offset=900000` Postgres must walk 900k tuples to discard them. The `ix_audit_events_ts` index becomes useless past the first few pages. **Opaque base64-encoded composite cursors** (`?cursor=eyJ0cyI6...`) hide implementation detail and survive sort-key changes, but Kestrel's audit table is sorted on a single indexed column (`ts`) that is naturally human-readable; the opacity buys nothing over an ISO timestamp and makes manual operator debugging harder. The chosen cursor is `?before_ts=<iso8601>&limit=N`: the query becomes `WHERE ts < :before_ts ORDER BY ts DESC LIMIT :limit`, which the `ix_audit_events_ts` index serves in O(log n + limit). Response shape is `{events: [...], next_before_ts: <iso> | null}` where `next_before_ts` is the `ts` of the last returned row when `len(events) == limit`, and `null` otherwise (signals last page). Limit is clamped server-side to `[1, 500]` (default 50) so callers can't ask for 1M rows. First-page request omits `before_ts` entirely — the absent param means "from now backwards."

### 7.6-revoke-semantics — `DELETE /admin/keys/{id}` is idempotent (already-revoked → 204); unknown id → 404; disambiguated by list-and-scan

The `PostgresApiKeyStore.revoke(key_id)` method (slice 1, slot 25) returns a bare `bool`: `True` when it flipped an active key to revoked, `False` for **both** "id unknown" and "id already revoked." The admin DELETE route needs to map those onto two different HTTP statuses: already-revoked → idempotent `204` success, genuinely-unknown → `404`. So the bool alone is insufficient; the route disambiguates. Three approaches were considered. **Trust the bool — `False` → 404** was rejected because it makes the endpoint non-idempotent: a client retrying a revoke after a network blip (it doesn't know the first attempt landed) would get a spurious `404` for a key it successfully revoked, which reads as "the key vanished" rather than "already done." Idempotency is the whole point of modeling revoke as a state transition rather than a deletion. **Add `store.get(key_id) -> ApiKeyInfo | None` to the `ApiKeyStore` Protocol** and check existence before revoking was rejected as contract creep — it widens a Protocol that *every* implementation must satisfy to serve one caller's rare error branch, and it costs a round-trip on the *common* path (every revoke would get+revoke). **Revoke-first, then `list()`-and-scan on miss** is the chosen path: call `revoke()`; if it returns `True`, done (`204`); if `False`, call `store.list()` and check whether the id is present — absent → `404`, present → it was already revoked → idempotent `204`. The extra query happens only on the miss path (uncommon), the common active-key revoke skips it entirely, and the Protocol is untouched. The success body is an empty `204 No Content` (conventional for DELETE; the caller already knows the id). The `404` is **not** audited — consistent with `7.3-emit-locus`, which accepts 404 audit gaps and relies on `kestrel_http_requests_total{status="404"}`. The `list()`-at-scale concern (full read on miss) is a non-issue at Kestrel's expected key counts; if a deployment ever has thousands of keys with frequent revoke-misses, that's the moment to add the deferred `get(id)` — revoke latency is the trigger.

### 7.6-key-mutation-backend — `POST`/`DELETE /admin/keys` return 503 when no key store is configured; `POST` returns 201 with the plaintext token exactly once

Both mutation routes require a real, writable store. When `api_key_backend == "null"` (the default — `get_api_key_store` returns `None` per `7-key-null-store-as-none`), there is nothing to write to. The routes raise **HTTP 503 Service Unavailable** with a plain message rather than pretending — minting a fake token would hand the caller an unusable credential they believe is real; silently no-op'ing a revoke would falsely report success. 503 mirrors exactly what `GET /admin/audit` already returns when no engine is built (slice 1), keeping the "endpoint is real, backend isn't wired up here" shape consistent across the admin surface. `POST` returns **201 Created** (not 200 — a new resource was made) and its body is `ApiKeyCreateResponse`, which is `ApiKeyResponse` plus a `token` field carrying the plaintext. That token is shown in this one response and nowhere else: the store keeps only `sha256(token)` (`7-key-hash`), so it is unrecoverable afterward, and no read endpoint (`GET /admin/keys`, `ApiKeyListResponse`) has a `token` field to leak one — the "shown once" rule is enforced structurally by the schema, not just by convention. Both routes audit *the caller's* identity (via `audit_id_for`) on the success path, answering "who performed this management action"; the created/revoked key's own id is recoverable from the action itself (the create logs it; the revoke's path template carries it).

## Phase 8 Decisions (Polish & Ship)

### 8-scope — six substeps; reinstate the blog post + SECURITY.md finalize the ROADMAP summary dropped

Brief §6 (p.13) lists nine Phase 8 deliverables; the `ROADMAP.md` one-line summary silently dropped two of them — "Blog post explaining the architecture" and "Security documentation finalized." Both are back in scope. Phase 8 is sequenced into six substeps, ordered so that each downstream deliverable can lean on an upstream one: **(1)** API service image + one-command Docker Compose (api + redis + postgres, migrations auto-run); **(2)** Python SDK; **(3)** integration examples built on the SDK; **(4)** documentation suite (README rewrite + architecture diagram, `docs/api.md`, `docs/architecture.md` + `DESIGN.md`, `docs/deployment.md`, finalize `SECURITY.md` + `CONTRIBUTING.md`, refresh `CHANGELOG.md` — retires all Phase-7 carry-in doc debt); **(5)** publish & deploy prep + blog draft (artifacts + commands handed off per `8-externals`); **(6)** closeout + tags (`8-version`). The image and Compose come first because the demo deploy, the Docker Hub publish, and the README quickstart all reference them. Brief §10's anti-scope-creep list (no multi-language, no GPU, no K8s, no gVisor/Firecracker, no web UI) stays binding throughout — Phase 8 is polish and packaging, not new capability.

### 8-api-image — API ships as its own image that drives the host Docker daemon (docker-out-of-docker); shared spool dir fixes the bind-mount path

There is no Dockerfile for the Kestrel API service today — `docker/executor/Dockerfile` only builds the sandbox *runtime* image. Phase 8 adds `docker/api/Dockerfile` (python:3.11-slim base + the Docker CLI binary + the `kestrel` package). The API container mounts the host Docker socket (`-v /var/run/docker.sock:/var/run/docker.sock`) and spawns sandbox containers on the host daemon — "docker-out-of-docker." This does **not** weaken the threat model: the API is the *trusted control plane* (Brief §2 — "Execution Manager creates/manages Docker containers"), while the *sandbox* containers it launches keep every §3 restriction including no socket access. Docker-in-Docker (privileged child daemon) was rejected as heavier and less secure than DooD for a single-node deployment (Brief §10 forbids clustering/K8s anyway). **The bind-mount consequence:** `DockerExecutor.run` writes user code to a `tempfile.TemporaryDirectory` and passes `--volume {host_path}:/sandbox/main.py:ro`; under DooD, `docker run --volume` resolves the source against the *host* filesystem, not the API container's, so a tempfile under the container's `/tmp` would mount as empty and the sandbox would run with no code. Resolution: a shared spool directory bind-mounted at the *same absolute path* into the API container, with the executor's tempfile base made configurable (new `KESTREL_EXEC_SPOOL_DIR`, default = system temp so host-run dev + the test suite are unchanged) and pointed at the spool when containerized. This preserves the Phase 2 lock (code is still a bind-mounted read-only tempfile, not stdin) — only the tempfile's location moves. Sessions are unaffected (stdin JSON-line delivery, no bind mount); rich-output files return via the kernel's stdout protocol, also no host mount. The runtime image must pre-exist on the host daemon (the API shells out to `docker run kestrel-runtime:0.5.0`), so Compose documents a `docker build` prestep for it rather than treating it as a normal long-running service.

### 8-sdk-scope — sync and async client covering the full API surface

The Python SDK ships both a synchronous and an asynchronous client, covering the entire public surface: `POST /execute`, the five `/sessions/*` lifecycle routes, the WebSocket streaming endpoint, and the HTTP polling fallback. Built on `httpx` (sync + async HTTP) and `websockets`. Sync-only was rejected because Kestrel's headline feature is real-time streaming and an SDK that can't demonstrate it undersells the product; async-only was rejected because the most common quick-start usage (a script, a notebook cell) is synchronous and shouldn't force users into an event loop. HTTP-only (drop streaming) was rejected for the same headline-feature reason. Admin endpoints are out of the SDK's first cut — they target operators who already have the `kestrel-keys` CLI; the SDK targets agent/application developers.

### 8-sdk-loc — SDK lives in-repo at clients/python/ as a separately-packaged distribution

The SDK lives in the same Git repo at `clients/python/` but is its **own** Python distribution with its own `pyproject.toml`, published to PyPI as `kestrel-client` (distinct from the service's `kestrel` package). Reasons: (a) the service and the client have disjoint dependency sets — the client must never drag in `docker`/`sqlalchemy`/`alembic`/`asyncpg`; (b) Brief §11's success criteria call for a "PyPI version" badge, which needs a real, independently-versioned published package; (c) co-locating in one repo keeps the client and server contract in lockstep without a second repo to sync. Folding the client into the service's existing `pyproject.toml` as a second package was rejected because it couples release cadence and dependency surface. A separate repo was rejected as premature for a solo learning project — monorepo is simpler to keep coherent.

### 8-docs-canonical — docs/architecture.md is the single architecture source; DESIGN.md points to it

Brief §5 lists both a root `DESIGN.md` and a `docs/architecture.md`, which invites two drifting copies of the same architecture narrative. Resolution: `docs/architecture.md` is the canonical, detailed architecture document (component diagram, request flow, the layered design, the key decision index pointing at this file); the root `DESIGN.md` is reduced to a short pointer to it. One source of truth, no divergence. The same principle applies to the threat model: `SECURITY.md` is canonical and the README's security section summarizes-and-links rather than restating.

### 8-externals — externally-gated deliverables are prepared as complete artifacts and handed off for the user to execute

Five Phase 8 deliverables require the user's own accounts or third-party actions: publishing the image to Docker Hub, deploying the public demo (Railway/Fly.io), publishing the blog post, and the inherently-external success criteria (≥3 outside testers; "Docker pulls"/"PyPI version" badges that only become real after publishing). For all of these the repository ships the complete artifact — image build recipe + exact `docker push` commands, deploy config + provider commands, a blog draft, badge markup — and the operator executes the publish/deploy with their own credentials. The repository prepares everything; the operator runs anything that touches external state. A substep is considered done when the artifact is ready, not gated on the external action.

### 8-version — v0.8.0 at phase close, then v1.0.0 (core shipped), plus the v0.2.0 backfill

At Phase 8 close, tag `v0.8.0` for the phase ship (consistent with the one-tag-per-phase rhythm: v0.1.0 … v0.7.0). Then cut `v1.0.0` to mark "the core is shipped" — the project scope frames v1.0 as the moment all eight phases are implemented, tested, documented, and the out-of-scope list becomes the post-v1.0 backlog. Also backfill the long-deferred `v0.2.0` annotated tag at `df2d45d` (the Phase 2 ship commit) so the tag history has no gap.

### 8-sdk-models — response shapes are plain frozen dataclasses, not pydantic

The SDK mirrors the server's response shapes (`ExecuteResult`, `Session`, `SessionExecuteResult`, the rich-output union, the stream-message union, `PollingRead`) as **frozen `@dataclass` types with `from_dict` classmethods**, not pydantic models. The point is to keep `kestrel-client`'s dependency footprint to just `httpx` + `websockets` — a client library that dragged in pydantic (and transitively pydantic-core) would impose a heavy, version-sensitive dependency on every consumer for what is, on the client side, simple JSON-to-object mapping. Discriminated unions (rich outputs by `type`, stream messages by `type`) are parsed by small hand-written dispatch functions. ISO-8601 timestamps are parsed with `datetime.fromisoformat`, with a one-line `Z`→`+00:00` normalisation so the same code works on Python 3.9–3.10 (whose `fromisoformat` predates `Z` support) as well as 3.11+. The server keeps using pydantic (it needs validation at the trust boundary); the client doesn't (it trusts its own server's responses).

### 8-sdk-errors — execution outcomes are data; only HTTP-level failures raise

The client faithfully reproduces Kestrel's "timeout is data, not an error" philosophy (see the Phase 1 timeout-strategy lock). A completed execute whose code timed out (`timed_out=True`) or exited non-zero (`exit_code != 0`) is returned as a normal result object — the SDK does **not** raise on it. The caller inspects the result. Exceptions are raised only for *HTTP-transport-level* failures, mapped from status codes to a typed hierarchy: `AuthenticationError` (401), `SessionNotFoundError` (404), `SessionBusyError` (409), `SessionGoneError` (410), `RateLimitedError` (429, carrying `retry_after`), and a catch-all `KestrelAPIError` (other 4xx/5xx, carrying `status_code` + `detail`), all under a `KestrelError` base. Rejected: raising on non-zero exit / timeout (would force callers into try/except for ordinary program output and break parity with the server contract); returning raw `httpx.Response` (pushes status-code interpretation onto every caller).

### 8-sdk-stream-transport — async `stream()` uses WebSocket; sync `stream()` uses the polling fallback

Both clients expose a `stream(session_id, code)` generator, but they ride different transports — deliberately, and matching what each runtime can do cleanly. `AsyncKestrelClient.stream()` is an async generator over the **WebSocket** endpoint (`websockets` is async-native; this is the low-latency path). `KestrelClient.stream()` (sync) is a generator built on the **HTTP long-poll fallback** (`POST …/execute/polling` then repeated `GET …/executions/{id}?since=&wait=`), yielding messages as they accumulate until `done`. This is a natural fit, not a compromise: synchronous callers are exactly the "WebSocket-blocked / simple HTTP only" audience the polling endpoint (Brief §6, decision 6.6-mech) was built for, and driving a WebSocket from synchronous code would mean hiding an event loop inside the sync client. Both clients also expose the raw polling primitives (`start_polling` / `read_execution`) for callers who want manual cursor control. Rejected: WS-only streaming (leaves sync users unable to stream without hand-rolling polling); running an event loop inside the sync client to use WS (surprising, fragile).

### 8-sdk-tests — unit tests via `httpx.MockTransport`; real interop proven by a manual live smoke

The committed SDK tests use `httpx.MockTransport` (and its ASGI/WebSocket equivalents) to assert the client builds the right requests (path, method, bearer header, JSON body) and parses responses into the right objects — deterministic, fast, and requiring no live server, mirroring how the service's own tests drive an in-process `TestClient`. Real end-to-end interop (the client talking to a genuinely running Kestrel over a real socket) is proven by a **manual live smoke against the Compose stack**, run during development and documented like `scripts/try_execute.py`, rather than committed as a test that would require standing up the full stack in CI. Rejected: committed integration tests requiring a live API (heavy infra dependency for every test run); mocking at the `KestrelClient` method level (would test nothing about the actual HTTP the client emits).

## Post-v1.0 Enhancements

Additive capabilities layered on after the v1.0.0 ship (all eight phases complete). Each keeps Kestrel general-purpose — no consumer-specific assumptions — and is backward-compatible by construction.

### exec-per-request-timeout — `POST /execute` accepts an optional `timeout_seconds`, clamped down to the server ceiling

Kestrel's wall-clock kill budget was process-global: `execute_timeout_seconds` (default 5s), fixed at server start, applied to every `/execute` call. That is too rigid for a general-purpose sandbox serving heterogeneous workloads — a caller running a quick snippet wants a tight fail-fast budget, while a caller running a heavier job wants longer, and one global value forces the operator to pick a single compromise or run a dedicated instance per workload class. Resolution: `ExecuteRequest` gains an optional `timeout_seconds`. The semantics are deliberately one-directional — `execute_timeout_seconds` becomes the **default** (applied when the field is omitted) *and* the **hard ceiling**: a supplied value is clamped *down* to it (`min(requested, ceiling)`) at the API boundary in `routes.py`. A request may therefore ask for a *shorter* budget but can never exceed the operator-configured maximum. This mirrors the per-request clamp Kestrel already uses for the polling long-poll `?wait=` parameter (`polling_max_wait_seconds`, decision 6.6-mech) — same shape, same safety property, so it is idiomatic rather than novel surface. Raising the maximum stays an operator decision (the env var), which is what keeps this a safe multi-tenant control rather than a way for any caller to pin a sandbox container open.

The change is additive at every layer: omitting the field (every pre-existing caller, the entire prior test suite) reproduces the previous behaviour byte-for-byte; the `Executor.run` Protocol and both backends (`SubprocessExecutor`, `DockerExecutor`) gain a `timeout_seconds: float | None = None` parameter that falls back to `settings.execute_timeout_seconds`, so any direct executor caller is unaffected; `ExecuteResponse` is unchanged (`duration_ms` + `timed_out` already reveal what happened). The clamp lives in the route (policy boundary) while the executors merely consume the resolved value (mechanism). The `kestrel-client` SDK exposes the field as a keyword-only `execute(code, *, timeout_seconds=None)` on both the sync and async clients, omitting it from the JSON body when `None` so the emitted request is identical to before.

Rejected alternatives: a second `execute_max_timeout_seconds` setting so per-request values could exceed the default up to a separate maximum (two-knob model) — more expressive but adds config surface for a capability the single-knob clamp already covers, since an operator who wants longer runs simply raises the one ceiling; allowing per-request values to *exceed* the ceiling (unbounded, or with only a soft cap) — breaks the resource guarantee that makes the sandbox safe to expose publicly; bundling a per-request output-cap override into the same change — deferred as unneeded for now, the identical clamp pattern can add it later if a real need appears.
